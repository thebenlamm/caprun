## §1. The bug, precisely

`crates/brokerd/src/server.rs`'s `run_broker_server` accept loop (lines 98-121) binds one abstract-namespace UDS socket per session and, for every accepted connection, spawns a fresh `handle_connection` task (`tokio::spawn` at line 106, closing over per-call clones of `conn`, `workspace_root`, `initial_last_event_hash`, and `initial_session_status`). `handle_connection` (starting line 134) then seeds three pieces of session-trust state as **per-connection locals**, freshly initialized on every single call:

- `session_status` (line 148): `let mut session_status = initial_session_status;` — a clone of whatever status the session had at broker-startup time, **not** a live lookup against the session's current row.
- `intent_provided` (line 153) and `fd_requested` (line 154): both `let mut ... = false;` — the Phase-16 BLOCKER-1 guard(a) ordering state, reset to `false` on every connection.

Guard(a) itself, at line 904 (`if *intent_provided || *fd_requested { ... reject ... }` inside the `ProvideIntent` arm), reads these per-connection locals by `&mut` reference threaded through `dispatch_request`'s parameter list (lines 239-245) — it never resolves them from IPC or from the database. The module doc (lines 129-133) and the per-arm comments (lines 145-154, 227-232) state this discipline explicitly: "never re-derived from IPC" — but the state itself is never re-derived from anything **shared**, either. It is local to the `tokio::spawn`ed task for that one connection.

**The falsified assumption:** the whole I0/I1/I2 model (`CLAUDE.md` "Security invariants", `PROJECT.md` Core Value) rests on `UserTrusted` meaning "the human typed it." Guard(a) was built (Phase 16, BLOCKER-1) specifically to make that hold: `ProvideIntent` is accepted exactly once and only before any `RequestFd`, so a worker cannot first read untrusted bytes and then mint an attacker-chosen literal as `UserTrusted`. That guarantee is real **within one connection**. It does not extend **across** connections, because nothing stops a second participant — the same worker process, or any other process that can reach the abstract socket — from opening a second `AF_UNIX` connection to the identical `\0/agentos/{session_id}` socket and getting a pristine `intent_provided = false`, `fd_requested = false`, and a stale (pre-demotion) `session_status`.

**Reachability is real, not theoretical.** Confinement never unshares the network namespace (`sandbox/lib.rs` — rlimits + Landlock + seccomp only); seccomp denies only `socket(AF_INET/AF_INET6)`, leaving `AF_UNIX` permitted (`seccomp.rs:6,64-83`); abstract-namespace UDS is explicitly unaffected by Landlock filesystem restrictions (`landlock.rs:4`, and this module's own doc header at lines 15-17: "Abstract sockets bypass Landlock filesystem restrictions. After Landlock deny-all-filesystem is applied in the worker, it can still connect to the broker's abstract socket."). A kernel-confined worker retains exactly the capability needed to mount this attack.

**Evidence anchor:** `crates/brokerd/tests/two_connection_intent_bypass.rs` is the reproduced live exfil. Its Part A is an intra-connection control proving guard(a) works normally (RequestFd then ProvideIntent on the *same* connection is rejected). Its Part B is the actual repro (lines 205-287): connection #1 calls `RequestFd` (tripping `fd_requested` on conn#1, simulating "the worker read the untrusted doc"), then a **fresh** connection #2 to the same socket calls `ProvideIntent` with an attacker-controlled literal (`"attacker@evil.com"`) — pristine guard(a) state and a stale `Active` `session_status` let it mint `UserTrusted` and, if routed to `email.send`, get `Allowed`. The test is `#[ignore]`d today ("documented failing regression... un-ignore when v1.4 Phase 0 fixes broker trust coherence") — it is the DONE gate this fix must flip green, by fixing the broker, never by weakening its safe-outcome assertions (lines 294-323).

**This test only covers the OVERLAPPING variant (conn#1 stays open, per its own line 221 comment).** Per the one-way-latch ruling above (§2), Phase 19 MUST additionally write a SEQUENTIAL-reconnect variant of this same regression test: conn#1 calls `RequestFd` and then cleanly disconnects (drops the stream) BEFORE conn#2 attempts to connect and call `ProvideIntent`. The safe-outcome assertion is identical (conn#2's `ProvideIntent` must be rejected, or if somehow accepted, the routed decision must not be `Allowed`) — but a fix implementation that only defeats the overlapping case (e.g., release-on-disconnect) would pass the existing test while leaving this sequential variant exploitable, so both must go green before Phase 19 is DONE.

**Structural note for Phase 19 (DESIGN-GATE-RECORD-v1.4.md Round 2, finding F2 — clarity, not a gate blocker):** the existing `two_connection_intent_bypass.rs` test connects Part A's `conn_a`, then Part B's `conn1`/`conn2`, ALL to the same `run_broker_server` invocation/socket, in sequence. Under the one-way latch, Part A's `conn_a` alone trips the latch for that broker instance's entire lifetime — so Part B's `conn1` would receive a rejection instead of `FdGranted` if run against the same broker instance, which is a **setup-assertion** failure (line 217-220's `FdGranted` expectation), not a safe-outcome assertion. Phase 19 MUST restructure this test file so each variant (Part A's guard(a) intra-connection control, the overlapping-connection repro, and the new sequential-reconnect repro) spawns its OWN fresh `run_broker_server` instance on its own socket name, so each variant's latch starts unset. Adjusting the setup-side `FdGranted` wiring to give each variant a fresh broker is NOT "weakening the safe-outcome assertions" (§1's DONE-gate language above) — the safe-outcome assertions themselves (Part C, lines 294-323) are untouched; only the test's connection topology is reorganized so each variant is tested against a correctly-scoped broker instance.

## §2. Fix shape: reject a 2nd connection to an already-active session (DESIGN-01)

**CHOSEN fix:** the broker refuses (rejects/closes) any new connection attempt to a session that already has an established, live connection. A confined worker legitimately holds **exactly one** connection to its session for the whole run — `run_broker_server`'s own doc (lines 62-68) describes "Accepts connections in a loop and spawns a per-connection task," but nothing in the v0-through-v1.3 design ever required or exercised more than one live connection per session; `N > 1` has no legitimate use case in shipped scope. Rejecting the 2nd connection closes the exact bypass in §1 with the smallest possible TCB change: single-connection-per-session enforcement, decided once, at accept time, before any per-connection trust state is ever seeded.

**Where the gate sits, precisely:** an authoritative "is this session's connection slot already occupied" check MUST be resolved **inside the accept loop** (`run_broker_server`, lines 98-121), **before** `tokio::spawn` hands the accepted stream to `handle_connection` (line 106-120). Concretely: the accept loop needs one new piece of state shared across iterations of the loop itself (e.g., an `Arc<Mutex<bool>>` or `Arc<AtomicBool>` "connection established" flag, or equivalently a `tokio::sync::Mutex`-guarded slot holding at most one live connection's identity) that is set the moment a connection is accepted and is checked — and, if already set, causes the just-accepted stream to be answered with a rejection response and then dropped — **before** `handle_connection` is ever invoked for that stream. This is the load-bearing placement requirement: if the gate ran *inside* `handle_connection` after `session_status`/`intent_provided`/`fd_requested` are already seeded (lines 144-154), a 2nd connection would still momentarily exist with pristine per-connection locals, reopening exactly the race this fix exists to close. The gate must be a pre-condition on reaching `handle_connection` at all, not a check performed once inside it.

**What "already active" means — a ONE-WAY, session-lifetime latch, never released (revised after DESIGN-GATE-RECORD-v1.4.md Round 1, finding F1).** The authoritative state is "has ANY connection for this `session_id` EVER completed its accept, at any point in this broker process's lifetime" — not "is a connection currently open." The gate is a single latch set the first time a connection is accepted and **never cleared for the life of the `run_broker_server` invocation**, including after the established connection disconnects. A 2nd connection is rejected whether it arrives *while* the 1st connection is still open (the overlapping case in `two_connection_intent_bypass.rs`) or *after* the 1st connection has cleanly closed (the sequential case: conn#1 does `RequestFd` then disconnects, conn#2 then attempts `ProvideIntent`).

**Why release-on-disconnect is unsound, not merely unnecessary.** An earlier draft of this section proposed releasing the latch when `handle_connection` exits, to permit "a legitimate reconnect after a clean disconnect (e.g., a crashed worker restarting)." This is a genuine BLOCKER, not a convenience trade-off: the adversarial worker process that reads the untrusted document on conn#1 retains those bytes in its own process memory regardless of whether conn#1 is open or closed. If the broker releases the latch on disconnect, that same adversarial worker simply closes conn#1 (after `RequestFd` has already let it read the doc) and opens conn#2 — pristine `intent_provided=false`/`fd_requested=false` on conn#2 (server.rs:153-154) let `ProvideIntent{"attacker@evil.com"}` mint `UserTrusted` exactly as in the overlapping case, because guard(a)'s per-connection locals were never shared or preserved across the disconnect. This is byte-for-byte the §1 exfiltration path, merely serialized instead of overlapped — release-on-disconnect does not close it, it only requires the attacker to add one `close()` call to the exploit. The regression test that anchors this fix (`two_connection_intent_bypass.rs`) keeps conn#1 open throughout (line 221's comment: "Leave conn#1 open (a real worker keeps its primary connection)") and therefore only exercises the overlapping variant — a release-on-disconnect implementation would pass that test while shipping the sequential variant of the same hole.

**No legitimate reconnect need exists in shipped scope, so the one-way latch costs nothing.** §2's own "no legitimate N>1 need exists today" reasoning (below) already establishes that a confined worker opens exactly one connection and keeps it for the life of the run — `cli/caprun`'s orchestrator model is one worker, one session, one connection, one `run_broker_server` invocation per session (main.rs: the broker task and the worker subprocess are spawned together, `child.wait()` bounds the worker's lifetime, and the broker's accept loop runs "forever" only in the sense of not returning early — it has no legitimate 2nd connection to accept). "A crashed worker restarting" is not a feature this project ships in v0-through-v1.3 and is not required by any REQUIREMENTS.md entry for v1.4; inventing a reconnect affordance to accommodate a hypothetical future crash-recovery feature is exactly the kind of scope a security fix must not silently add. If crash-recovery reconnect is ever a real requirement, it needs the REJECTED shared-coherent-state alternative below (re-deriving trust state from the DB on reconnect) — which is precisely why permitting reconnect without that redesign is unsound. A future milestone that genuinely needs reconnect must reopen this decision explicitly with the user, not smuggle it in as a "release on disconnect" implementation detail.

**Implementation is simpler than the rejected draft, not more complex.** A one-way latch requires no release mechanism, no `Drop` guard, and no reasoning about *when* release is safe — it is a single write-once flag (e.g., a loop-local `bool` in `run_broker_server`'s own frame, set immediately after the first successful accept, checked before every subsequent accept's `tokio::spawn`; no `Arc`/`Mutex` cross-task sharing is even required since the accept loop itself is single-threaded per session). Phase 19 MUST NOT implement any release-on-disconnect path for this latch.

**What a rejected 2nd connection receives:** the broker MUST still complete the accept (so the caller does not merely see a TCP-like RST/connection-refused with no diagnosis) and then send a single framed `BrokerResponse::Error` (the existing error-response shape already used throughout `dispatch_request`, e.g. line 168-176's oversized-message rejection) naming the reason ("a connection is already established for this session; only one connection per session is permitted"), then close the stream without ever constructing a `handle_connection` task, a `ValueStore`, or seeding `session_status`/`intent_provided`/`fd_requested` for it. No audit event needs to be appended for a rejected 2nd connection (it never reached a state that could affect the session's trust posture) — Phase 19 may choose to log one for observability, but it is not a correctness requirement of this design.

**The REJECTED alternative: shared coherent multi-connection state.** The alternative considered and rejected is making `session_status`, `intent_provided`, and `fd_requested` **coherent across connections** instead of forbidding a 2nd connection at all — e.g., one authoritative per-session store (behind a single `Arc<Mutex<...>>` or resolved fresh from the `sessions` table on every access) that every concurrently-open connection reads and writes, so guard(a) and `session_status` agree no matter how many connections are open. This is explicitly **not** the chosen approach for v1.4. Reasons, stated in writing:

- **Larger blast radius.** Coherent shared state means every one of `intent_provided`, `fd_requested`, and `session_status` (and any future per-connection trust local) must be redesigned as shared, lock-protected, cross-task state, multiplying the surface that must be gotten right in the TCB. Rejecting a 2nd connection touches only the accept loop.
- **More TCB surface for no present need.** Concurrency-correctness work (lock ordering, contention, what "the ProvideIntent-once" guarantee even means when two connections could race to call it simultaneously) has to be designed and proven for a capability — genuine multi-connection operation on one session — that no shipped or currently-planned feature needs.
- **No legitimate N>1 need exists today.** A confined worker opens one connection and keeps it for the life of the run (`cli/caprun`'s orchestrator model: one worker, one session, one connection). Phase 20's forward-looking planner connection (§3 below) is a **second, capability-restricted connection to a *different* purpose**, not a second connection racing the worker's own trust state on the *same* mint-capable slot — see §3 for why the per-verb capability split, not shared coherent state, is the correct mechanism for that case.

This section makes the choice unambiguous: **reject-a-2nd-connection is the chosen fix; shared coherent multi-connection state is the rejected alternative**, named here specifically so Phase 19 does not reintroduce it as a "simpler" workaround if the reject-outright gate proves inconvenient during implementation. If a genuine N>1 need is ever discovered, that would require reopening this decision explicitly with the user (per `PROJECT.md`'s Locked Decisions discipline), not silently building shared state as a Phase 19 implementation detail.

**Implementability for Phase 19:** this fix is specified precisely enough against `server.rs`'s actual accept loop (lines 98-121) and per-connection locals (`session_status` at line 148; `intent_provided`/`fd_requested` at lines 153-154) that Phase 19 can implement it — add one per-session occupancy latch, set in the accept loop the first time a connection is accepted, checked before every subsequent accept's `tokio::spawn`, and **never released for the life of the `run_broker_server` invocation** (revised per the one-way-latch ruling above) — without touching the per-connection locals' own semantics or reintroducing the bug by moving the check somewhere those locals have already been seeded.

## §3. Per-verb capability split (DESIGN-05)

Forward-looking for Phase 20's planner connection (`PLANNER-02`): a connection must be able to hold **NO mint verb**. All three mint-capable `BrokerRequest` variants — `ProvideIntent` (server.rs ~894-1011), `ReportClaims` (~393-479), and `ReportDerivedClaim` (~795-892) — MUST be unavailable on a capability-restricted connection, independent of and in addition to §2's single-connection-per-session gate.

**A fourth arm also mints trust and must not be treated as a permitted planner verb: `CreateSession`** (server.rs lines 248-338) mints a fresh, forced-`Active` session via `SeedProvenance::TrustedArg` — but it is already gated fail-closed by guard-(c) (§4: `CAPRUN_ENABLE_IPC_CREATE_SESSION` exact-string `"1"` opt-in, default-deny). A planner connection hitting this arm in production gets the existing fail-closed `BrokerResponse::Error` regardless of this section's capability split. Phase 20's capability-set model should still enumerate `CreateSession` alongside the three mint verbs above when defining "what a planner connection may never do," so the capability set's default-deny reasoning does not implicitly assume guard-(c) will always be the only thing standing between a planner connection and this arm.

**Why this is a distinct mechanism from §2, not a substitute for it:** §2's reject-2nd-connection gate answers "how many connections may exist for one session" (answer: one, for the worker). This section answers a different question Phase 20 will need answered: "what may a given *permitted* connection do." Phase 20 intends to add a **second, distinct connection** — the planner's — to the same session, which is exactly the case §2 forbids for the worker's own connection. Reconciling this: §2's single-connection gate is scoped to *mint-capable* connections (the worker's). Phase 20 must extend the accept-time gate so that a connection requesting planner-only capabilities is accepted as a **second, capability-restricted** connection that never enables `ProvideIntent`/`ReportClaims`/`ReportDerivedClaim` in the first place — it is not a "2nd worker connection" in the sense §2 rejects, because it structurally cannot reach any mint site regardless of how many such connections exist. This document does not specify how Phase 20 signals "this connection wants planner capabilities" (that is Phase 20's own design work) — it only fixes the shape the capability check itself must take, so Phase 20 has a concrete target: extend the connection-establishment gate from a boolean ("occupied or not") to a **capability set** decision, not redesign trust-state coherence again.

**Shape of the capability model (design intent, not Rust):**
- A per-connection **capability set** is decided once, at connection-establishment time (the same accept-time decision point as §2's occupancy gate) — never re-derived per-message, mirroring the `session_status`/guard(a) discipline's own "resolved once, trusted-path-only" principle (`DESIGN-session-trust-state.md` §4).
- The capability set is checked at **each mint-verb dispatch arm** (`ProvideIntent`, `ReportClaims`, `ReportDerivedClaim` inside `dispatch_request`) — a connection whose capability set excludes mint verbs receives the same category of fail-closed `BrokerResponse::Error` rejection §2 specifies for a rejected 2nd connection, mints nothing, advances no chain head.
- **Fail-closed, default-deny:** the default capability set for any connection MUST NOT include mint verbs unless explicitly granted at establishment. This mirrors guard-(c)'s (§4 below) fail-closed opt-in discipline and the project's standing `CON-i2-non-bypassable` constraint — a capability model that defaults permissive would be exactly the self-declaration hole `DESIGN-session-trust-state.md` §4 already closed for `session_status`, reopened one layer up.
- Non-mint verbs (`RequestFd`, `SubmitPlanNode`) are unaffected by this split — a planner connection needs `SubmitPlanNode` (to route handles) but never needs to mint a handle itself; it only ever references `ValueId`s the worker's connection already minted. (Cross-connection `ValueId` resolution is already denied by HARD-03's per-connection `ValueStore` isolation — server.rs's module doc, lines 37-39 — so a planner connection referencing a worker-minted handle is a Phase 20 design question about how handles are shared, not a re-opening of HARD-03.)

Phase 20 (`PLANNER-02`) relies on exactly this: the planner connection holds no mint verb, so even a fully adversarial planner emitting arbitrary `PlanNode{sink, args}` values can only ever route handles it did not mint and cannot launder untrusted content into a `UserTrusted` handle by calling `ProvideIntent` itself — the tainted-vs-trusted determination stays entirely with the worker's mint sites (§5). This is designed now, even though §2's Phase-0 fix does not itself require a capability split (the worker's single connection legitimately needs all three mint verbs) — because Phase 20 needs a concrete target and a capability-set model composes cleanly with §2's occupancy gate (one gate, two orthogonal properties: "is this session's worker slot occupied" and "what may this specific connection do").

## §4. guard-(c) interaction (DESIGN-06)

The reject-2nd-connection fix (§2) does **not** widen `CAPRUN_ENABLE_IPC_CREATE_SESSION` (server.rs lines 267-283). That guard remains exactly as shipped: the `CreateSession` IPC arm checks `std::env::var("CAPRUN_ENABLE_IPC_CREATE_SESSION").as_deref() == Ok("1")` (line 267-270, exact-string-match, never `.is_ok()`) and, when unset/empty/anything-else, returns a fail-closed `BrokerResponse::Error` (lines 271-283) minting nothing. §2's fix operates entirely on the **accept loop and per-connection trust-state seeding** for an *existing* session's socket — it has no interaction with `CreateSession`, which mints a **new, session-independent** session (line 285's comment: "does NOT thread the connection chain") via a wholly separate code path. Nothing in §2 changes when guard-(c) fires, what it denies, or its default-deny posture.

**Re-stating whether guard-(c) should finally be compile-excluded:** this residual has stood since Phase 16 (`PROJECT.md`'s v1.3 residual-risks clause: "Guard-(c) ... is a runtime default-deny flag, not a compile-time exclusion — the forced-Active mint code ships in the production binary. v2: build-exclude it.") The case for compile-exclusion: shipping code in the production binary that can force a session `Active` via an env-var flip is strictly more attack surface than not shipping it at all — an operator misconfiguration (accidentally exporting `CAPRUN_ENABLE_IPC_CREATE_SESSION=1` in a production environment, e.g. via a leaked `.env` or an inherited shell) is a live risk the runtime check cannot fully close, since the confined worker cannot set the *broker process's* environment but an operator or a broader deployment environment still can. Compile-exclusion (e.g., `#[cfg(test)]` or a Cargo feature gated to test builds only) would remove the arm from the production binary entirely, closing that operator-misconfiguration channel structurally rather than by runtime check.

The case against doing it **now**, in v1.4 Phase 0: this fix's DONE gate is narrowly scoped to the cross-connection bypass (§1-§2) plus the forward-looking capability split (§3) and the replay/mint-site/oracle rulings (§5-§7) — `.planning/REQUIREMENTS.md`'s Out of Scope table already records "Guard-(c) compile-time exclusion — Re-confirmed, not re-scoped." Compile-excluding guard-(c) is orthogonal implementation work (likely a `cfg`/feature-flag change plus verifying every test call site that currently relies on the runtime-enabled arm still compiles under the test-only feature) that has no dependency on and no effect on the cross-connection fix itself; bundling it into Phase 19 would widen that phase's scope without a corresponding increase in risk-reduction urgency (the flag is already fail-closed by default; the residual is "what if an operator misconfigures the environment," not "this is exploitable by a confined worker today").

**Recommendation:** perform the compile-exclusion as a v1.5 (or later) hardening item, not v1.4 Phase 19 — same disposition `.planning/REQUIREMENTS.md` already recorded. This document re-confirms, per DESIGN-06's charge, that v1.4 does **not** commit to doing the compile-exclusion; it only re-states the question and gives a recommendation for a future milestone to pick up.

## §5. Three-mint-site audit (DESIGN-03)

The v1.4 scoping review's claim — "the planner can't launder what it never holds" — depends on a narrower, code-verified claim about which of `crates/brokerd`'s three mint sites can produce a `UserTrusted` handle directly from a supplied string. Auditing each against its actual dispatch arm:

- **`mint_from_read`** (`ReportClaims` arm, server.rs lines 393-479; mint call at line 428) — mints a genuinely-tainted `ValueRecord` per typed claim (`WorkerClaim::EmailAddress`/`RelativePath`/`DocFragment`, lines 405-423). This is safe **only via the sensitivity map**: `mint_from_read` taints purely by `claim_type` (line 401's comment: "the broker independently assigns the claim_type; the worker cannot launder trust"), stamping the resulting handle `ExternalUntrusted` (or an equivalent untrusted taint per `quarantine.rs`'s mint policy), which Blocks at every routing/content-sensitive sink arg per the executor's sensitivity map. It is **not** a site that yields a `UserTrusted` handle from a supplied string — its whole design intent is the opposite (produce a handle the executor will Block, not Allow, when routed to a sensitive sink).
- **`mint_from_intent`** (`ProvideIntent` arm, server.rs lines 894-1011; mint calls at lines 949, 962, 979) — mints the worker-supplied IPC literal (from `CaprunIntent::SendEmailSummary`'s `recipient`/`subject`/`body` or `CreateFileFromReport`'s `path`, lines 926-932) directly as `UserTrusted`. **This is the T1 hole**: unlike `mint_from_read`, there is no sensitivity-map interposition — the literal arrives over IPC from the worker process and is minted trusted on the strength of guard(a) alone (the "exactly once, only before any RequestFd" ordering check at line 904). §1-§2 establish that guard(a)'s per-connection scoping is exactly what a 2nd connection defeats. `mint_from_intent` **is** the one site that yields a TRUSTED handle from a supplied string.
- **`mint_from_derivation`** (`ReportDerivedClaim` arm, server.rs lines 795-892; mint call at line 849) — mints a transform-derived literal the worker computed from prior handles (lines 800-844: every `input_value_ids` entry is resolved against the connection's own `ValueStore` first, failing closed on any unresolved handle; `mint_from_derivation` itself performs byte-verify + every-input-`file_read`-rooted guards internally, per its own doc contract referenced at lines 839-844). It is safe via **byte-verify and provenance-rootedness**, not via sensitivity-map interposition (its inputs may already be untrusted, and their taint composes rather than being reset) — it is not a site that can mint a fresh `UserTrusted` handle from an arbitrary supplied string; it can only re-derive from handles the broker already resolved, verified against their own byte-exact provenance.

**The corrected, narrower claim (stated verbatim, per this section's charge):** *"the only mint yielding a TRUSTED handle from a supplied string is `ProvideIntent`."*

**The reopening tripwire, stated explicitly:** this reduction holds only while every sensitive sink's sensitivity map (`crates/executor/src/sink_sensitivity.rs`'s `is_routing_sensitive`/`is_content_sensitive`, and `DESIGN-session-trust-state.md` §6's `sink_effect_class`) treats `WorkerExtracted`/`ExternalUntrusted` taint as **blocking** on the relevant arg. It reopens the instant any future sink's sensitivity map treats either of those taint labels as *non-blocking* on some arg — at that point `mint_from_read` (or `mint_from_derivation`, transitively) would produce a handle that reaches an `Allowed` sink despite carrying untrusted taint, which is functionally equivalent to a second trusted-from-supplied-string mint site even though no code changed at the mint sites themselves. This is a **standing invariant** any future sink addition or sensitivity-map edit must preserve — it is not self-enforcing in the type system and must be checked by the reviewer/implementer at every future sink-sensitivity change, exactly as `DESIGN-session-trust-state.md` §10 already requires exhaustive-match discipline for `EffectClass`/`DenyReason` additions.

## §6. Replay risk re-earned under the adaptive-planner threat model (DESIGN-02)

**Ruling on MAJOR-2.** The Allowed `email.send` dispatch path (server.rs lines 705-788) has no CAS/idempotency mechanism: its own comment (lines 770-776) already names this — "REPLAY RESIDUAL RISK (named, not silent): this Allowed path has no CAS/PendingConfirmation — a replayed SubmitPlanNode mints a fresh effect_id and would send again (N submissions => N emails)." That risk was accepted for v1.3 under an "external replay actor" framing (a network-level or client-bug replay, not an adversary reasoning about the system). v1.4 introduces an **adaptive planner** (Phase 21) as the milestone's own named adversary — this changes who the replay actor plausibly is, and per `DOC-01`'s honesty discipline, that acceptance must be **re-earned in writing against the new actor**, not silently inherited from v1.3's framing.

**The new actor.** Under an adaptive planner, the replay actor collapses from "external" (a flaky client, a network retry, an unrelated bug) to **the milestone's own named adversary** — an adversarial planner that has already been shown (by design, per Phase 21's HARD GATE) to comply with hostile-document instructions and attempt to route tainted values to sensitive sinks. If that same planner (or the worker it's paired with) can also cause `SubmitPlanNode` to be resubmitted — whether by genuinely re-emitting the same plan node, or by a hostile instruction inducing a legitimate retry-on-error client loop to fire twice — the replay actor is no longer "assume good faith, bug aside" but "assume an adversary that benefits from duplication."

**Blast-radius analysis.** A replayed `SubmitPlanNode` for an **already-Allowed** decision (§5's narrower claim matters here: `Allowed` on `email.send` requires every arg to have passed the executor's per-arg taint check, meaning every arg's taint is *not* untrusted-and-sensitive) mints a fresh `effect_id` (server.rs line 484) and re-executes the SMTP send (lines 777-787) — this is duplication, not a *new* recipient or body. Critically: **an untrusted/tainted recipient or body still Blocks** (I2 is unchanged by replay — replay only duplicates an already-`Allowed` decision, it cannot convert a would-be-`Blocked` decision into `Allowed`). So replay amplification is **bounded to trusted/human-typed recipients** — the only inputs that can ever reach `Allowed` on `email.send` in the first place (per §5, only `ProvideIntent`, i.e. worker-IPC-supplied literals guarded by §2's fix, mints `UserTrusted`). The amplification an adaptive planner could induce via replay is therefore **DoS/duplication of a legitimate send**, not new exfiltration — the planner cannot use replay to cause an *untrusted* value to be sent that would otherwise have Blocked.

**The existing durable ledger.** Every `email.send` Allowed-dispatch appends a durable, per-attempt `email_send_attempted` event (server.rs lines 750-768) **before** the SMTP socket ever opens, parent-chained onto the just-advanced `plan_node_evaluated` head, under the same lock as the append. This makes every replayed send independently auditable — an operator or a future automated monitor can observe N `email_send_attempted` events for what should have been one send and detect the duplication after the fact, even though nothing currently prevents it before the fact.

**This bound is contingent on §2's one-way-latch fix actually shipping.** The claim "amplification is structurally bounded to trusted/human-typed recipients" depends entirely on §2's fix being sound — if the cross-connection bypass were to remain reachable (e.g., via the release-on-disconnect variant §2 rejects after Round 1 remediation), an attacker literal could reach `Allowed`, and replay would then amplify an attacker-controlled send rather than merely duplicating a legitimate one. This section's ruling holds only downstream of §2's ONE-WAY latch being correctly implemented in Phase 19 — it is not an independent guarantee.

**Re-earned ruling (v1.4, written explicitly, using the required language):** the replay risk on the Allowed `email.send` path is **re-earned as an Accepted Residual Risk for v1.4**, under the adaptive-planner threat model, because (a) amplification is structurally bounded to trusted/human-typed recipients — I2's per-arg Block is unaffected by replay and untrusted content still Blocks regardless of how many times a plan node is resubmitted, (b) the durable per-attempt ledger (lines 750-768) makes every send auditable, and (c) adding a CAS/idempotency mechanism is explicitly out of scope for this milestone (`.planning/REQUIREMENTS.md` Out of Scope: "CAS/idempotency token on the Allowed email.send path — Replay risk re-earned in writing... instead"). This document does **not** design a CAS mechanism. The **named v2 obligation**: a minimal CAS/idempotency check on the Allowed `email.send` path (already tracked in `.planning/todos/pending/2026-07-08-v1.3-phase16-v2-security-obligations.md`) remains open and is **not closed** by this re-earning — it is carried forward, now explicitly re-justified rather than silently inherited. Should Phase 21's planner ever gain a mint verb (which §3's capability split is designed specifically to prevent), this ruling would need to be revisited, per `PROJECT.md`'s own Key Decisions table entry for this exact contingency.

## §7. The decision is an oracle (DESIGN-04 / MEDIUM-1)

**The oracle, documented explicitly.** `ExecutorDecision::Allowed` vs `ExecutorDecision::BlockedPendingConfirmation { anchors: Vec<BlockedArg> }` (`executor_decision.rs` lines 184-207, specifically 185-202) is returned to whichever connection submitted the `SubmitPlanNode` request. Whoever observes that decision learns, at minimum, whether the submitted plan node's args carried untrusted-and-sensitive taint — a binary but security-relevant signal about the **handles that connection referenced**. **Scope of "referenced" today:** under HARD-03's per-connection `ValueStore` isolation (server.rs's module doc, lines 37-39, and the per-connection `ValueStore` seeded at line 144), a connection can only reference `ValueId`s it minted itself — a connection attempting to reference another connection's handle receives a fail-closed `Denied{DanglingHandle}`-class rejection, not a taint-revealing decision. So today this oracle is only reachable by a connection observing decisions about handles *it minted* — the cross-connection-observability risk this section exists to pre-empt becomes live only once Phase 20 introduces cross-connection handle sharing (§3's own note that "how handles are shared" is a Phase 20 design question), which is exactly why §7's reduced-signal ruling below is scoped to Phase 20's planner connection specifically. When the decision is `BlockedPendingConfirmation`, each element of `anchors: Vec<BlockedArg>` carries a `SinkBlockedAnchor` (lines 128-151) whose `literal_sha256` field (line 144) is, by its own doc comment (lines 138-143), "SHA-256 digest (lowercase hex) of the byte-exact literal — the DURABLE, tamper-evident representation." A recipient of the decision who already suspects a particular literal (e.g., "is the blocked recipient `attacker@evil.com`?") can hash that candidate string themselves and compare against the observed `literal_sha256` — this is an **offline literal-guess confirmer**: it does not require any further broker interaction to test a guess, and a match is a cryptographically strong confirmation (a mismatch is not fully dispositive of a wrong guess only in the negligible-probability SHA-256-collision sense). `BlockedArg.literal` itself (line 180, "the LIVE byte-exact literal ... carried in-memory for the confirmation UX") is explicitly documented as NOT part of the hashed hash chain — only `anchor.literal_sha256` is durable/tamper-evident — but a connection that receives the **full** `BlockedPendingConfirmation` value over IPC receives `anchors[i].literal` too, i.e., the plaintext literal itself, not merely its digest.

**The forward-looking ruling: does Phase 20's planner connection see the full decision, or a reduced signal?**

**Ruling: a reduced signal.** Phase 20's planner connection MUST NOT receive the full `Allowed`/`BlockedPendingConfirmation { anchors, .. }` decision shape verbatim over its `SubmitPlanNode` response. It should receive a signal sufficient only to let it proceed or stop attempting further plan nodes for the current turn — functionally an allow/block boolean (or a small enum distinguishing "proceed" from "blocked, do not retry this exact plan node") — **without** `anchors`, without `literal_sha256`, and without the plaintext `literal` field.

**Rationale, tied back to §3's capability model:** the planner needs only enough signal to know whether its emitted `PlanNode` succeeded or was blocked, so it can decide what to plan next (e.g., stop, or try a different sink) — it does not need, and per PLAN-03's compile-time-enforced discipline (`planner.rs`'s own doc header: "This module holds ONLY opaque `ValueId` handles — it NEVER sees: a `ValueRecord`... a raw byte slice or string from untrusted content... a taint label"), it must **structurally never see** the literal or its digest. Handing a capability-restricted planner connection the full anchor set — literal_sha256 confirmers for every blocked arg — would reopen exactly the oracle this section names, at the one seam (Phase 20's new connection) that is otherwise designed (§3) to hold no mint verb and thus should not need any taint-observability either. The reduction is consistent with §3's per-verb capability split: a connection whose capability set excludes mint verbs should likewise receive a reduced decision signal, not the full TCB-internal anchor/digest detail that only a worker connection (which already holds the literals in its own `ValueStore`) has any legitimate need to see. Phase 20 owns the exact reduced-signal shape (a new, smaller response variant, or a projection of the existing one) — this document only rules that the reduction must happen and states why.

**Residual for connections that DO see full decisions:** the worker's own connection legitimately receives the full decision (it already holds every literal in its own `ValueStore` — the confirmation UX depends on this, per `DESIGN-confirmation-release.md`'s "verbatim recipient+body narration"). This oracle is therefore an accepted residual for the worker connection specifically (it is not new exposure — the worker already has the literal), and is closed for the planner connection specifically by this section's reduced-signal ruling. Recorded in §9.

## §9. Accepted Residual Risks

**1. Replay on the Allowed `email.send` path (no CAS) — re-earned for v1.4, not newly accepted.** See §6. Bounded to duplication/DoS of trusted, human-typed sends; untrusted content still Blocks regardless of replay. The durable `email_send_attempted` per-attempt ledger (server.rs lines 750-768) makes every send auditable after the fact but does not prevent duplication before it. v2 obligation: minimal CAS/idempotency check on this path (tracked in `.planning/todos/pending/2026-07-08-v1.3-phase16-v2-security-obligations.md`).

**2. The three-mint-site reduction's reopening tripwire.** See §5. "Only `ProvideIntent` yields a TRUSTED handle from a supplied string" holds only while every sensitive sink's sensitivity map treats `WorkerExtracted`/`ExternalUntrusted` taint as blocking on the relevant arg. Any future sink addition or sensitivity-map edit that relaxes this for some arg silently reopens a second trusted-from-supplied-string-equivalent path. This is a standing invariant, not self-enforcing in the type system — future sink/sensitivity-map changes must be checked against it explicitly.

**3. The decision oracle for the worker's own connection.** See §7. The worker connection legitimately receives the full `Allowed`/`BlockedPendingConfirmation{anchors, literal_sha256, ...}` decision (it already holds every literal in its own `ValueStore`; the confirm/deny UX depends on this). This is not new exposure introduced by this fix — it predates v1.4 — and is explicitly NOT extended to Phase 20's planner connection, which receives a reduced signal per §7's ruling. Recorded here so a future change does not accidentally widen the worker connection's own exposure further (e.g., by exposing the full decision to some third connection type not yet designed) without re-examining this section.

**4. Guard-(c) remains a runtime flag, not a compile-time exclusion.** See §4. `CAPRUN_ENABLE_IPC_CREATE_SESSION` continues shipping in the production binary as a fail-closed, exact-string-match, default-deny runtime opt-in. This fix does not widen it and does not compile-exclude it. Recommendation given (§4) for a future milestone; v1.4 does not commit to performing the compile-exclusion.

**5. T2 (slot-type binding) — OUT OF SCOPE, deferred to v1.5, NOT designed in this document.** The executor does not check that a handle's semantic origin (its taint/trust label) matches the semantic role of the slot it is routed into (e.g., a `UserTrusted` handle placed in a `to` slot is neither sensitive-untrusted nor slot-checked, so I2 does not fire on that basis alone). This is safe today only *incidentally* — every `UserTrusted` handle is human-typed (via `ProvideIntent`, and after this fix, coherently guarded across connections per §2), so a misrouted handle carries the human's own string rather than an attacker's. `.planning/PROJECT.md`'s Key Decisions table and `.planning/REQUIREMENTS.md`'s Out of Scope table both already record this deferral (Ben, 2026-07-10 scoping): "keeps v1.4 to one milestone; enforcing it now would split v1.4 into two milestones." This document explicitly does not design slot-type binding enforcement — no new `DenyReason` variant, no slot/taint-matching logic is specified here. It is carried forward as v1.5 scope.
