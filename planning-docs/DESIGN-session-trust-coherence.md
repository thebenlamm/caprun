## §1. The bug, precisely

`crates/brokerd/src/server.rs`'s `run_broker_server` accept loop (lines 98-121) binds one abstract-namespace UDS socket per session and, for every accepted connection, spawns a fresh `handle_connection` task (`tokio::spawn` at line 106, closing over per-call clones of `conn`, `workspace_root`, `initial_last_event_hash`, and `initial_session_status`). `handle_connection` (starting line 134) then seeds three pieces of session-trust state as **per-connection locals**, freshly initialized on every single call:

- `session_status` (line 148): `let mut session_status = initial_session_status;` — a clone of whatever status the session had at broker-startup time, **not** a live lookup against the session's current row.
- `intent_provided` (line 153) and `fd_requested` (line 154): both `let mut ... = false;` — the Phase-16 BLOCKER-1 guard(a) ordering state, reset to `false` on every connection.

Guard(a) itself, at line 904 (`if *intent_provided || *fd_requested { ... reject ... }` inside the `ProvideIntent` arm), reads these per-connection locals by `&mut` reference threaded through `dispatch_request`'s parameter list (lines 239-245) — it never resolves them from IPC or from the database. The module doc (lines 129-133) and the per-arm comments (lines 145-154, 227-232) state this discipline explicitly: "never re-derived from IPC" — but the state itself is never re-derived from anything **shared**, either. It is local to the `tokio::spawn`ed task for that one connection.

**The falsified assumption:** the whole I0/I1/I2 model (`CLAUDE.md` "Security invariants", `PROJECT.md` Core Value) rests on `UserTrusted` meaning "the human typed it." Guard(a) was built (Phase 16, BLOCKER-1) specifically to make that hold: `ProvideIntent` is accepted exactly once and only before any `RequestFd`, so a worker cannot first read untrusted bytes and then mint an attacker-chosen literal as `UserTrusted`. That guarantee is real **within one connection**. It does not extend **across** connections, because nothing stops a second participant — the same worker process, or any other process that can reach the abstract socket — from opening a second `AF_UNIX` connection to the identical `\0/agentos/{session_id}` socket and getting a pristine `intent_provided = false`, `fd_requested = false`, and a stale (pre-demotion) `session_status`.

**Reachability is real, not theoretical.** Confinement never unshares the network namespace (`sandbox/lib.rs` — rlimits + Landlock + seccomp only); seccomp denies only `socket(AF_INET/AF_INET6)`, leaving `AF_UNIX` permitted (`seccomp.rs:6,64-83`); abstract-namespace UDS is explicitly unaffected by Landlock filesystem restrictions (`landlock.rs:4`, and this module's own doc header at lines 15-17: "Abstract sockets bypass Landlock filesystem restrictions. After Landlock deny-all-filesystem is applied in the worker, it can still connect to the broker's abstract socket."). A kernel-confined worker retains exactly the capability needed to mount this attack.

**Evidence anchor:** `crates/brokerd/tests/two_connection_intent_bypass.rs` is the reproduced live exfil. Its Part A is an intra-connection control proving guard(a) works normally (RequestFd then ProvideIntent on the *same* connection is rejected). Its Part B is the actual repro (lines 205-287): connection #1 calls `RequestFd` (tripping `fd_requested` on conn#1, simulating "the worker read the untrusted doc"), then a **fresh** connection #2 to the same socket calls `ProvideIntent` with an attacker-controlled literal (`"attacker@evil.com"`) — pristine guard(a) state and a stale `Active` `session_status` let it mint `UserTrusted` and, if routed to `email.send`, get `Allowed`. The test is `#[ignore]`d today ("documented failing regression... un-ignore when v1.4 Phase 0 fixes broker trust coherence") — it is the DONE gate this fix must flip green, by fixing the broker, never by weakening its safe-outcome assertions (lines 294-323).

## §2. Fix shape: reject a 2nd connection to an already-active session (DESIGN-01)

**CHOSEN fix:** the broker refuses (rejects/closes) any new connection attempt to a session that already has an established, live connection. A confined worker legitimately holds **exactly one** connection to its session for the whole run — `run_broker_server`'s own doc (lines 62-68) describes "Accepts connections in a loop and spawns a per-connection task," but nothing in the v0-through-v1.3 design ever required or exercised more than one live connection per session; `N > 1` has no legitimate use case in shipped scope. Rejecting the 2nd connection closes the exact bypass in §1 with the smallest possible TCB change: single-connection-per-session enforcement, decided once, at accept time, before any per-connection trust state is ever seeded.

**Where the gate sits, precisely:** an authoritative "is this session's connection slot already occupied" check MUST be resolved **inside the accept loop** (`run_broker_server`, lines 98-121), **before** `tokio::spawn` hands the accepted stream to `handle_connection` (line 106-120). Concretely: the accept loop needs one new piece of state shared across iterations of the loop itself (e.g., an `Arc<Mutex<bool>>` or `Arc<AtomicBool>` "connection established" flag, or equivalently a `tokio::sync::Mutex`-guarded slot holding at most one live connection's identity) that is set the moment a connection is accepted and is checked — and, if already set, causes the just-accepted stream to be answered with a rejection response and then dropped — **before** `handle_connection` is ever invoked for that stream. This is the load-bearing placement requirement: if the gate ran *inside* `handle_connection` after `session_status`/`intent_provided`/`fd_requested` are already seeded (lines 144-154), a 2nd connection would still momentarily exist with pristine per-connection locals, reopening exactly the race this fix exists to close. The gate must be a pre-condition on reaching `handle_connection` at all, not a check performed once inside it.

**What "already active" means:** the authoritative state is "has any connection for this `session_id` completed its accept and not yet disconnected" — not a duration, not a message count, not `session_status` (which is an orthogonal I1/I0 concept `DESIGN-session-trust-state.md` already owns). The flag/slot must be released (so a legitimate reconnect after a clean disconnect remains possible — e.g., a crashed worker restarting) when the established connection's `handle_connection` loop exits (its `Ok(())` / `Err(e)` return in `run_broker_server`'s `tokio::spawn` block, lines 106-120) — Phase 19 must decide the exact release mechanism (e.g., a `Drop` guard, or an explicit clear after `handle_connection`'s future resolves), but the release MUST happen only on genuine disconnect, never speculatively or on a timer, so a live worker's single connection is never evicted out from under it.

**What a rejected 2nd connection receives:** the broker MUST still complete the accept (so the caller does not merely see a TCP-like RST/connection-refused with no diagnosis) and then send a single framed `BrokerResponse::Error` (the existing error-response shape already used throughout `dispatch_request`, e.g. line 168-176's oversized-message rejection) naming the reason ("a connection is already established for this session; only one connection per session is permitted"), then close the stream without ever constructing a `handle_connection` task, a `ValueStore`, or seeding `session_status`/`intent_provided`/`fd_requested` for it. No audit event needs to be appended for a rejected 2nd connection (it never reached a state that could affect the session's trust posture) — Phase 19 may choose to log one for observability, but it is not a correctness requirement of this design.

**The REJECTED alternative: shared coherent multi-connection state.** The alternative considered and rejected is making `session_status`, `intent_provided`, and `fd_requested` **coherent across connections** instead of forbidding a 2nd connection at all — e.g., one authoritative per-session store (behind a single `Arc<Mutex<...>>` or resolved fresh from the `sessions` table on every access) that every concurrently-open connection reads and writes, so guard(a) and `session_status` agree no matter how many connections are open. This is explicitly **not** the chosen approach for v1.4. Reasons, stated in writing:

- **Larger blast radius.** Coherent shared state means every one of `intent_provided`, `fd_requested`, and `session_status` (and any future per-connection trust local) must be redesigned as shared, lock-protected, cross-task state, multiplying the surface that must be gotten right in the TCB. Rejecting a 2nd connection touches only the accept loop.
- **More TCB surface for no present need.** Concurrency-correctness work (lock ordering, contention, what "the ProvideIntent-once" guarantee even means when two connections could race to call it simultaneously) has to be designed and proven for a capability — genuine multi-connection operation on one session — that no shipped or currently-planned feature needs.
- **No legitimate N>1 need exists today.** A confined worker opens one connection and keeps it for the life of the run (`cli/caprun`'s orchestrator model: one worker, one session, one connection). Phase 20's forward-looking planner connection (§3 below) is a **second, capability-restricted connection to a *different* purpose**, not a second connection racing the worker's own trust state on the *same* mint-capable slot — see §3 for why the per-verb capability split, not shared coherent state, is the correct mechanism for that case.

This section makes the choice unambiguous: **reject-a-2nd-connection is the chosen fix; shared coherent multi-connection state is the rejected alternative**, named here specifically so Phase 19 does not reintroduce it as a "simpler" workaround if the reject-outright gate proves inconvenient during implementation. If a genuine N>1 need is ever discovered, that would require reopening this decision explicitly with the user (per `PROJECT.md`'s Locked Decisions discipline), not silently building shared state as a Phase 19 implementation detail.

**Implementability for Phase 19:** this fix is specified precisely enough against `server.rs`'s actual accept loop (lines 98-121) and per-connection locals (`session_status` at line 148; `intent_provided`/`fd_requested` at lines 153-154) that Phase 19 can implement it — add one shared per-session occupancy gate, checked and set in the accept loop before `handle_connection` is spawned, released on disconnect — without touching the per-connection locals' own semantics or reintroducing the bug by moving the check somewhere those locals have already been seeded.

## §3. Per-verb capability split (DESIGN-05)

Forward-looking for Phase 20's planner connection (`PLANNER-02`): a connection must be able to hold **NO mint verb**. All three mint-capable `BrokerRequest` variants — `ProvideIntent` (server.rs ~894-1011), `ReportClaims` (~393-479), and `ReportDerivedClaim` (~795-892) — MUST be unavailable on a capability-restricted connection, independent of and in addition to §2's single-connection-per-session gate.

**Why this is a distinct mechanism from §2, not a substitute for it:** §2's reject-2nd-connection gate answers "how many connections may exist for one session" (answer: one, for the worker). This section answers a different question Phase 20 will need answered: "what may a given *permitted* connection do." Phase 20 intends to add a **second, distinct connection** — the planner's — to the same session, which is exactly the case §2 forbids for the worker's own connection. Reconciling this: §2's single-connection gate is scoped to *mint-capable* connections (the worker's). Phase 20 must extend the accept-time gate so that a connection requesting planner-only capabilities is accepted as a **second, capability-restricted** connection that never enables `ProvideIntent`/`ReportClaims`/`ReportDerivedClaim` in the first place — it is not a "2nd worker connection" in the sense §2 rejects, because it structurally cannot reach any mint site regardless of how many such connections exist. This document does not specify how Phase 20 signals "this connection wants planner capabilities" (that is Phase 20's own design work) — it only fixes the shape the capability check itself must take, so Phase 20 has a concrete target: extend the connection-establishment gate from a boolean ("occupied or not") to a **capability set** decision, not redesign trust-state coherence again.

**Shape of the capability model (design intent, not Rust):**
- A per-connection **capability set** is decided once, at connection-establishment time (the same accept-time decision point as §2's occupancy gate) — never re-derived per-message, mirroring the `session_status`/guard(a) discipline's own "resolved once, trusted-path-only" principle (`DESIGN-session-trust-state.md` §4).
- The capability set is checked at **each mint-verb dispatch arm** (`ProvideIntent`, `ReportClaims`, `ReportDerivedClaim` inside `dispatch_request`) — a connection whose capability set excludes mint verbs receives the same category of fail-closed `BrokerResponse::Error` rejection §2 specifies for a rejected 2nd connection, mints nothing, advances no chain head.
- **Fail-closed, default-deny:** the default capability set for any connection MUST NOT include mint verbs unless explicitly granted at establishment. This mirrors guard-(c)'s (§4 below) fail-closed opt-in discipline and the project's standing `CON-i2-non-bypassable` constraint — a capability model that defaults permissive would be exactly the self-declaration hole `DESIGN-session-trust-state.md` §4 already closed for `session_status`, reopened one layer up.
- Non-mint verbs (`RequestFd`, `SubmitPlanNode`) are unaffected by this split — a planner connection needs `SubmitPlanNode` (to route handles) but never needs to mint a handle itself; it only ever references `ValueId`s the worker's connection already minted. (Cross-connection `ValueId` resolution is already denied by HARD-03's per-connection `ValueStore` isolation — server.rs's module doc, lines 37-39 — so a planner connection referencing a worker-minted handle is a Phase 20 design question about how handles are shared, not a re-opening of HARD-03.)

Phase 20 (`PLANNER-02`) relies on exactly this: the planner connection holds no mint verb, so even a fully adversarial planner emitting arbitrary `PlanNode{sink, args}` values can only ever route handles it did not mint and cannot launder untrusted content into a `UserTrusted` handle by calling `ProvideIntent` itself — the tainted-vs-trusted determination stays entirely with the worker's mint sites (§5). This is designed now, even though §2's Phase-0 fix does not itself require a capability split (the worker's single connection legitimately needs all three mint verbs) — because Phase 20 needs a concrete target and a capability-set model composes cleanly with §2's occupancy gate (one gate, two orthogonal properties: "is this session's worker slot occupied" and "what may this specific connection do").

## §4. guard-(c) interaction (DESIGN-06)

The reject-2nd-connection fix (§2) does **not** widen `CAPRUN_ENABLE_IPC_CREATE_SESSION` (server.rs lines 267-283). That guard remains exactly as shipped: the `CreateSession` IPC arm checks `std::env::var("CAPRUN_ENABLE_IPC_CREATE_SESSION").as_deref() == Ok("1")` (line 267-270, exact-string-match, never `.is_ok()`) and, when unset/empty/anything-else, returns a fail-closed `BrokerResponse::Error` (lines 271-283) minting nothing. §2's fix operates entirely on the **accept loop and per-connection trust-state seeding** for an *existing* session's socket — it has no interaction with `CreateSession`, which mints a **new, session-independent** session (line 285's comment: "does NOT thread the connection chain") via a wholly separate code path. Nothing in §2 changes when guard-(c) fires, what it denies, or its default-deny posture.

**Re-stating whether guard-(c) should finally be compile-excluded:** this residual has stood since Phase 16 (`PROJECT.md`'s v1.3 residual-risks clause: "Guard-(c) ... is a runtime default-deny flag, not a compile-time exclusion — the forced-Active mint code ships in the production binary. v2: build-exclude it.") The case for compile-exclusion: shipping code in the production binary that can force a session `Active` via an env-var flip is strictly more attack surface than not shipping it at all — an operator misconfiguration (accidentally exporting `CAPRUN_ENABLE_IPC_CREATE_SESSION=1` in a production environment, e.g. via a leaked `.env` or an inherited shell) is a live risk the runtime check cannot fully close, since the confined worker cannot set the *broker process's* environment but an operator or a broader deployment environment still can. Compile-exclusion (e.g., `#[cfg(test)]` or a Cargo feature gated to test builds only) would remove the arm from the production binary entirely, closing that operator-misconfiguration channel structurally rather than by runtime check.

The case against doing it **now**, in v1.4 Phase 0: this fix's DONE gate is narrowly scoped to the cross-connection bypass (§1-§2) plus the forward-looking capability split (§3) and the replay/mint-site/oracle rulings (§5-§7) — `.planning/REQUIREMENTS.md`'s Out of Scope table already records "Guard-(c) compile-time exclusion — Re-confirmed, not re-scoped." Compile-excluding guard-(c) is orthogonal implementation work (likely a `cfg`/feature-flag change plus verifying every test call site that currently relies on the runtime-enabled arm still compiles under the test-only feature) that has no dependency on and no effect on the cross-connection fix itself; bundling it into Phase 19 would widen that phase's scope without a corresponding increase in risk-reduction urgency (the flag is already fail-closed by default; the residual is "what if an operator misconfigures the environment," not "this is exploitable by a confined worker today").

**Recommendation:** perform the compile-exclusion as a v1.5 (or later) hardening item, not v1.4 Phase 19 — same disposition `.planning/REQUIREMENTS.md` already recorded. This document re-confirms, per DESIGN-06's charge, that v1.4 does **not** commit to doing the compile-exclusion; it only re-states the question and gives a recommendation for a future milestone to pick up.
