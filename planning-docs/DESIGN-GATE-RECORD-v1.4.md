## Revision History

- **Round 1 — BLOCKED** (fresh, independent, code-tracing reviewer; not the Plan 18-01 authoring session). Found:
  1. **BLOCKER (F1) — release-on-disconnect + permitted-reconnect reopens the exact bypass sequentially.** The original §2 design set the occupancy latch at accept but released it when the established connection's `handle_connection` loop exited, explicitly to permit "a legitimate reconnect after a clean disconnect (e.g., a crashed worker restarting)." The reviewer traced that an adversarial worker process retains the untrusted document's bytes in its own memory independent of connection state: it can call `RequestFd` on conn#1, close conn#1, then open conn#2 and call `ProvideIntent` with the attacker literal — conn#2 gets pristine `intent_provided=false`/`fd_requested=false` (server.rs:153-154) under the release-on-disconnect design, exactly as in the overlapping-connection case, just serialized. The existing regression test (`two_connection_intent_bypass.rs`, conn#1 kept open per its own line-221 comment) only exercises the overlapping variant and would go green under an implementation that leaves the sequential variant exploitable. *Resolution:* §2 rewritten to specify a ONE-WAY, session-lifetime latch — set on first accept, never released for the life of the `run_broker_server` invocation, no reconnect permitted at all. Verified this stays within the LOCKED "reject a 2nd connection" decision (it is, if anything, a smaller/simpler mechanism than the rejected draft — no release logic, no `Drop` guard, no cross-task `Arc`/`Mutex` needed). Added an explicit "why release-on-disconnect is unsound, not merely unnecessary" subsection, and required Phase 19 to add a SEQUENTIAL-reconnect regression test variant alongside the existing overlapping-connection test.
  2. **MINOR (F3) — §7 oracle overstated cross-connection reach.** The doc's phrasing implied a connection could observe taint signal about handles it never minted. The reviewer confirmed HARD-03's per-connection `ValueStore` isolation (server.rs:37-39, 144) means a connection referencing another connection's handle gets a fail-closed `Denied` rejection, not a taint-revealing decision — so the oracle is reachable today only for handles a connection minted itself; the cross-connection version of this risk only becomes live once Phase 20 introduces handle sharing. *Resolution:* §7 amended with a "Scope of 'referenced' today" clarification tying the oracle's current reach to HARD-03 and explicitly deferring the wider concern to Phase 20's reduced-signal design (which §7 already specifies).
  3. **MINOR (F4) — §3's mint-verb enumeration omitted `CreateSession`.** `CreateSession` (server.rs:248-338) also mints trust (a forced-`Active` session) but is separately gated fail-closed by guard-(c). *Resolution:* §3 amended with a note that Phase 20's capability-set model should enumerate `CreateSession` alongside the three named mint verbs, so the "no mint verb" default-deny reasoning does not implicitly assume guard-(c) alone will always cover it.
  4. **(Noted, not a separate finding requiring edit) §6's replay bound is contingent on §2.** The reviewer observed that §6's "amplification bounded to trusted/human-typed recipients" claim depends entirely on §2's fix being sound. *Resolution:* §6 amended with an explicit dependency note — this ruling holds only downstream of §2's one-way latch being correctly implemented in Phase 19, not as an independent guarantee.

  All 4 items resolved by direct edit to `DESIGN-session-trust-coherence.md`; no finding was resolved by weakening a ruling or by designing a locked-out mechanism (no shared multi-connection state, no CAS, no slot-type binding introduced).
