## 1. The reframe

`PROJECT.md:472` defers to v1.4+ the question: *"does taint survive a real LLM planner's regeneration (laundering)?"*

**Proposal: that is the wrong question, and the code already says so.**

Laundering is unpatchable at the token level. A model that re-emits a tainted value as fresh model-authored tokens has genuinely destroyed provenance. There is no mechanism that recovers it. Chasing "taint survives regeneration" is a research program with no floor.

But `crates/runtime-core/src/plan_node.rs:101-110` already shipped the answer:

```rust
/// A planner-facing sink argument. Binds a name to an opaque `ValueId`.
/// Carries NO literal and NO taint field — that is the whole point of the handle
/// model (T-04-02 mitigation). An injected planner that fabricates a `PlanArg`
/// still cannot strip taint or forge a literal: those live in the broker-owned
/// `ValueRecord` keyed by `value_id`.
pub struct PlanArg { pub name: String, pub value_id: ValueId }
```

Plus `DenyReason::DanglingHandle` (`executor_decision.rs:18`): a forged or dangling handle **never becomes `Allowed`.**

**The planner cannot launder what it never holds.** v1.1 shipped session-scoped handles (HARD-03) and a handles-only planner (HARD-02). `PLAN.md:31` already specifies the Tier-3 hard split: *"privileged planner sees typed extracts only."*

So v1.4 is not research. It is **wiring an LLM into a socket that was designed for it**, and then proving the socket holds.

## 2. What v1.4 must actually prove

**HARD GATE (proposed):** A real LLM planner, given a document containing a prompt injection that explicitly instructs it to email attacker@evil.com, cannot cause an email to be sent to attacker@evil.com — **because it never possessed the literal string `attacker@evil.com`**, only an opaque `ValueId` whose broker-owned `ValueRecord` is taint-marked read-rooted.

The demo must show, in one live Linux run:
1. The LLM planner emits a syntactically valid `PlanNode` with `PlanArg{name:"to", value_id: <tainted handle>}` — i.e. **the model complies with the injection**
2. The executor Blocks it deterministically, `verify_chain` true, Mailpit count == 0
3. A trusted-intent control on the same sink Allows and delivers exactly once
4. **The model's full context is dumped and grep'd: `attacker@evil.com` does not appear in it**

Point 4 is the milestone. The others re-prove v1.3 with a new planner.

## 3. The threat that actually matters (and it isn't laundering)

Phase 16 found and closed a live exfiltration hole: *"a worker could skip `ReportClaims` and inject an arbitrary recipient via `ProvideIntent`."*

`ProvideIntent` → `mint_from_intent` mints a **UserTrusted** `ValueRecord` from a caller-supplied string. That is a legitimate trust root: the human typed it.

**With a deterministic extractor, that hole was closed by three guards. With an LLM planner in the loop, the adversary behind those guards gets much smarter.**

The v1.4 threat model, ranked:

| # | Threat | Why it's new with an LLM |
|---|--------|--------------------------|
| **T1** | **Trusted-mint forgery.** The planner calls `ProvideIntent("attacker@evil.com")`, minting a UserTrusted handle for an attacker-controlled literal it read from the doc. | This is laundering — but through the *IPC surface*, not the token stream. The model retypes what it read. Guards from Phase 16 must be re-proven against an adversary that can reason about them. **This is the milestone's #1 risk.** |
| **T2** | **Confused-deputy slot assignment.** The planner cannot forge or strip, but it *chooses which handle goes in which named slot*. Trusted-subject handle → `to` slot. | Deterministic extractor never made a choice. An LLM does. Is name-binding in `combined_digest` sufficient? |
| **T3** | **Covert channel via handle selection.** Handle *choice* is planner-controlled and low-bandwidth-but-nonzero. Selecting handle A vs B over N sinks leaks bits of the literal. | Only exploitable by a model that can plan across calls. Probably accept-and-document, not solve. |
| **T4** | **Typed-extract poisoning.** The planner sees "typed extracts only" (`PLAN.md:31`) — but if an extract's *summary text* carries attacker instructions, the model is steered even though it can't carry the literal. | I1 already answers this (steering ≠ authority). Blocked at the sink regardless. Prove it, don't redesign it. |

**T1 is the whole milestone.** If an LLM planner can mint a trusted handle for a literal it read from an untrusted doc, every other invariant is decoration. The deterministic extractor could not do this *because it had no capacity to retype what it read*. A language model's entire function is retyping what it read.

**Candidate mitigation (for reviewers to attack):** `ProvideIntent` must not be reachable from the planner at all. Intent minting is a **human-only trust root**, bound to the CLI invocation, sealed before the planner's first token. If the planner needs a value the human didn't type, the answer is "it doesn't get one." This may make the clean-allow path require the human to have typed the recipient — which is arguably *correct* and arguably guts the demo. Reviewers: which?

## 4. Proposed scope

**IN:**
- A minimal LLM planner behind the existing `Planner` trait surface, emitting `PlanNode{sink, args: Vec<PlanArg>}` and nothing else. Model choice irrelevant; use the cheapest thing that follows a tool schema.
- Planner runs **outside** the confined worker, sees typed extracts + handle IDs, holds no caps, no network beyond its own inference endpoint.
- Context-dump assertion harness (gate item 4 above).
- T1 guard hardening + live negative control: an LLM explicitly instructed by an injected doc to `ProvideIntent` the attacker recipient, proven unable to.
- T2 negative control: LLM instructed to place the trusted-subject handle into `to`.
- Design gate first (project rule): `DESIGN-llm-planner-boundary.md` before any code.

**OUT (say so loudly):**
- Taint surviving regeneration. Declared unpatchable-by-construction; superseded by "planner never holds the literal." **If reviewers disagree, this whole scope is wrong.**
- Steganographic / covert-channel elimination (T3) — document as accepted residual, consistent with v0's existing steg acceptance (`PROJECT.md:332`).
- Multi-turn planner memory, planner self-correction loops, any planner that is not stateless per PlanNode.
- Model-quality claims of any kind. This proves a *boundary*, not a model.

**Explicitly NOT reopened:** the v1.3 residual risks (forgeable chain head, Allowed-path replay, guard-(c) runtime-vs-compile-time). They remain accepted. v1.4 does not touch them; if anything an LLM planner makes the replay gap *worse* and that should be re-stated, not silently inherited.

## 5. Why this is the right next milestone

Four independent signals, 2026-06-01 → 06-24, all post-dating v1.2:
- **"Prompt injection as role confusion"** (06-24) — LLMs receive everything as one undifferentiated token stream with no structural separation between internal thought and external speech. An argument that **in-band defenses cannot work**. This is caprun's out-of-band TCB thesis, derived independently.
- **MosaicLeaks** (HF/ServiceNow, 06-19) — deep-research agents combining private docs with web retrieval routinely leak sensitive information. This is v1.3's hero demo, restated as a research finding.
- **Workday** (06-01) — "the AI agent bottleneck is permissions, not model capability."
- **Google DeepMind AI Control Roadmap** (06-19) — system-level security "even if alignment is perfect."

Everything the market shipped in response is **containment, not taint**: LangSmith microVM sandboxes, Vercel Connect scoped credentials, NVIDIA SkillSpector pre-install scanning, OpenAI Lockdown Mode. None propagate provenance to a sink argument and block deterministically.

**I2 is unoccupied.** But the only thing that would make anyone believe it is a demo where the adversary is a real language model, not a `for` loop. Right now caprun has never met the client it exists to defend against.
