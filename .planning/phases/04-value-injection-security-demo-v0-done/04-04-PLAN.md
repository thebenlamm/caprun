<objective>
Build the quarantined reader extraction path and the genuine-taint mint anchor (REQ-quarantined-reader), and wire the broker effect path to the deterministic executor. This is the single highest-risk plan: it is where taint becomes genuine. Closes RESEARCH Gap 3 (dispatch wiring) and Gap 7 (reader uses pre-confinement file open, no broker fd-pass needed for §9).

Purpose: The §9 acceptance contract fails if taint is stapled at the sink. The only defense is that the broker code path which records the read Event is the same path that mints the value record, with provenance_chain anchored at that read Event's id. mint_from_read is that anchor and must be the sole construction site of a tainted ValueRecord in the broker.
Output: A quarantine module (deterministic extractor + mint_from_read anchor), brokerd depending on the executor crate, the SubmitPlanNode dispatch arm calling executor::submit_plan_node, and brokerd::submit_plan_node delegating to the executor — all green.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| hostile file → confined reader | The reader opens the file before confinement and reads under apply_confinement (Phase 3 boundary); only a typed lossy claim crosses outward. |
| reader → broker (mint) | The broker's mint_from_read sets taint from the read Event; the reader/planner never authors taint. |
| planner → broker (SubmitPlanNode) | The dispatch arm resolves authoritative records from the trusted store via the executor — planner-supplied metadata is never trusted. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-04-03 | Spoofing | mint_from_read (genuine-taint anchor) | critical | mitigate | The same function appends the file_read Event and mints the record with provenance_chain=[read_event.id]; stapled taint is impossible because no other broker path mints a tainted record. Asserted by the anchor-identity test. |
| T-04-01 | Tampering | SubmitPlanNode dispatch | critical | mitigate | The broker effect path calls the deterministic executor against the trusted store; there is no raw EffectRequest→sink bypass (negative grep on NotImplemented). |
| T-04-04 | Elevation of Privilege | confined reader | high | mitigate | The reader holds no send/net/exec capability and runs under sandbox::apply_confinement (proven by Phase 3 negative assertions 29/29); it emits only a typed lossy claim, stripping instructional content. |
| T-04-SC | Tampering | cargo dependencies | low | accept | Adds only an in-workspace `executor` path dependency; no new registry crate. extract_email_claims uses existing workspace deps (no new pattern crate). |
