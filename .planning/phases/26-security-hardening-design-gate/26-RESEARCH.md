## User Constraints (from CONTEXT.md)

### Locked Decisions

**HARDEN-01 — demote-at-RequestFd (I1 honest scope) → §a**
- **D-01 (principle, locked):** Worker-reported evidence may only DEMOTE a session, never keep it Active; any "keep-Active" decision must be BROKER-derived. Demotion commits **before** the fd is released to the worker (fail-closed ordering).
- **D-02 (pinned-doc reconciliation, locked):** The pinned clause `DESIGN-session-trust-state.md:80-81` ("No other function in brokerd MUST be permitted to set `SessionStatus::Draft` for the I1 reason") is reconcilable, not blocking — the status quo actually violates its own anti-self-declaration rationale, because `mint_from_read` (the only demotion site today) is reached solely via the worker-optional `ReportClaims` path, so a silent/injected worker skips demotion entirely. Relocating the demotion to fd-grant is the broker's own act, which strengthens the spirit. Precedent: `fd_requested` is flipped broker-side at RequestFd entry today (`server.rs:1001`). The design doc must state this reconciliation explicitly and amend the pinned doc's letter to name RequestFd as a broker-side demotion site.
- **D-03 (CONTROL-01 clean path, locked):** The benign fragment-free doc read must still stay Active and send. Its stay-Active criterion shifts from "fragment-free" to "trusted-labeled file" — the design doc must specify where the trusted label at fd-grant time comes from (broker-derived), and the fail-closed default for an unlabeled file is demote. Avoid dragging content-parsing into the TCB at fd time (TOCTOU risk) — the label must be broker-derivable without parsing worker-supplied bytes.

**HARDEN-02 — authenticated audit chain (verify_chain) → §b**
- **D-04 (threat ceiling, OWNER RULING = "In-host DB-writer"):** v1.6 defends against an actor with `events`-table write access on the host — NOT a full host/root compromise. Mechanism: a keyed MAC over the chain with the key held by the broker, outside the confined worker's Landlock filesystem scope, so a bare DB-file writer cannot derive it. Add an anchored/monotonic chain head to catch tail-truncation and restore-from-backup rollback (a per-entry MAC alone does NOT detect either).
- **D-05 (deferred, named residual):** A full host/root compromise that can read the broker's key is explicitly out of scope for v1.6 and must be recorded as a named Accepted Residual Risk (candidate: external out-of-store notarization in a later milestone). Do NOT let the doc claim tamper-evidence beyond the in-host-DB-writer model.
- **D-06 (mutable-field trap, locked):** The current audit design keeps one field mutable/redactable outside the hash. Under a MAC, that field would be wholly unauthenticated — the design doc must either accept that explicitly or fold a salted commitment of it inside the MAC'd payload. (Verify against `audit.rs` before pinning.)

**HARDEN-03 — Allowed-path replay CAS → §c (largely mechanical)**
- **D-07 (mechanism, locked):** An idempotency key / compare-and-swap on the trusted (Allowed) `email.send` path, in the same atomic-transaction discipline as the confirm path's PendingConfirmation at-most-once.
- **D-08 (scope caveat — MUST be in the doc, locked):** CAS buys at-most-once PER PLAN NODE, NOT bounded sends per session. A statically-compromised worker that mints fresh, distinct plan nodes still sends N times — no plan-layer dedup defeats that. The doc must state this limit and file an effects-budget / rate-limit as named future defense-in-depth (not v1.6 scope). Pin what the idempotency key is derived from (plan-node hash vs client nonce) during research.

**HARDEN-04 — compile-out the forced-Active mint → §d (largely mechanical)**
- **D-09 (mechanism, locked):** Replace the runtime `CAPRUN_ENABLE_IPC_CREATE_SESSION` env gate (`server.rs:918-932`) with a dedicated Cargo feature (e.g. `test-fixtures`) so the forced-`Active` `CreateSession` arm is physically absent from a default release build. Note the known complication: `#[cfg(test)]` is NOT set when `brokerd` compiles as a dependency of an integration-test binary — which is why the runtime flag was chosen originally; the feature-flag approach is what closes it.
- **D-10 (verification-integrity gate, locked):** Because Cargo unifies features, a plain `cargo test` builds the lib WITH the feature. The mitigation therefore needs its own negative gate — a separate no-feature build asserting the IPC arm is absent (grep of the built artifact / a compile-fail or symbol-absence check) — else it verifies nothing. Also confirm Phase 30's live proof runs a featureless binary matching the shipped release.

**HARDEN-05 — file.create `contents` slot → §e**
- **D-11 (mechanism, RECOMMENDED depth taken — owner did not object):** Give `contents` a content-sensitive I2 treatment (mirror email `body`), so a tainted value routed into it is blocked/confirmed under the same I2/slot-type discipline as other sensitive args. Today it is `None` (unconstrained) at `sink_sensitivity.rs:157`. Pin the expected-role list carefully to avoid false-positive blocking of legitimate derived file writes (existing trusted-content flows must still succeed — ROADMAP SC-3).
- **D-12 (laundering residual, tracked as cross-cutting):** The real deeper risk is a write→re-read laundering round-trip (worker writes tainted `contents` to a workspace file, then re-reads it via RequestFd with provenance evaporated). Full output-file provenance labeling would close it directly but is a bigger, adapter-spanning change. For v1.6 the input-role treatment is the scoped fix; the laundering loop is closed by the label-continuity ruling (X-01) below and recorded as a tracked residual.

**Cross-Cutting rulings (the design doc MUST pin these — no single residual names them)**
- **X-01 (label continuity):** Broker-written files must carry the writing session's taint label, or demote-at-RequestFd (§a) is bypassable via the (§e) laundering loop. ONE label-provenance model answers both HARDEN-01 and HARDEN-05.
- **X-02 (shared-store restart authority):** Session status, taint labels, PendingConfirmation rows, and the new CAS rows all live in the same SQLite store beside the audit chain — a DB-writer can flip `SessionStatus` or delete a CAS row, and the change goes live at broker restart. The doc must pin ONE rule: which security state is authoritative in-memory vs recovered from disk, and on restart recovered state is either MAC-re-verified (ties to HARDEN-02) or fail-closed re-derived (e.g. all recovered sessions resume `Draft`).
- **X-03 (TOCTOU / atomic ordering):** A demotion racing an in-flight Allowed `SubmitPlanNode` must be safe — pin ONE uniform atomicity rule: the status transition commits before effect dispatch (same discipline as demote-before-fd-release in D-01).

## Summary

All five residuals trace to code this session (not carried from CONTEXT.md's prior pass) and every one of the five locked mechanism DIRECTIONS in CONTEXT.md is confirmed consistent with the current source — no correction is needed to D-01/D-02/D-04/D-07/D-09/D-11 themselves. However, three concrete groundings surface material, previously-unstated risk the design doc must address, and one is a genuinely NEW finding not named by any existing locked decision:

1. **HARDEN-05's biggest risk is not "no known-safe role vocabulary" (as CONTEXT.md frames Assumption A2) — it is that the ONLY production value ever bound to `file.create`'s `contents` arg today is the reused `path`-role `intent_value_id`** (`cli/caprun/src/planner.rs:200-209`; `server.rs:1312` mints it with `origin_role: "path"`). `CreateFileFromReport`'s worker path never extracts or reports any doc-derived "contents" claim at all (`cli/caprun/src/worker.rs:293-312` only ever reports `RelativePath` claims). If the design doc pins `contents`' expected-role list to anything that does NOT include `"path"`, Phase 29 will hard-Deny the ONLY existing file.create flow — a direct regression of ROADMAP Phase-26 SC and the existing `s9_live_file_create_clean_allow` test. This must be pinned explicitly, not left as "TBD, Phase 29's to pin" (CONTEXT.md currently defers the exact membership to "Claude's Discretion" — this research narrows that discretion to one concrete, load-bearing fact).
2. **HARDEN-01's "trusted-labeled file" signal (D-03) has no existing broker-side plumbing to derive from.** The one candidate broker-derivable-without-parsing-bytes signal — "does the RequestFd'd path match the CLI's own `<workspace-file>` argument" — is real and cheap, but that value (`workspace_rel`, `cli/caprun/src/main.rs:187-189`) is currently forwarded ONLY to the worker process via the `WORKSPACE_FILE` env var, never to the broker/`run_broker_server`. Phase 27 needs a genuinely new plumbing hop (thread `workspace_rel` into `run_broker_server`/`dispatch_request` as new broker-owned state, mirroring how `workspace_root`/`session_status` are already threaded) — not merely a new `if` branch inside the existing `RequestFd` arm.
3. **A previously-undocumented cross-connection staleness gap exists in `session_status` today, independent of and more severe than `DESIGN-session-trust-state.md` §12's "Accepted Residual Risk #1."** Every connection accepted by `run_broker_server` — including the Planner-role connection introduced in v1.4 Phase 20 — seeds its own per-connection `session_status` local from the SAME `initial_session_status` function parameter cloned at the top of the accept loop (`server.rs:202`, `:231`), which is the session's status AT SESSION-CREATION time, never re-read from the `sessions` table. If the Worker connection's `ReportClaims` demotes the session to `Draft` (updating the DB row) AFTER a Planner connection has already been accepted — which is the live v1.4/v1.5 architecture's normal order (worker reads untrusted content, then hands routing to the planner) — the Planner connection's own `SubmitPlanNode` calls still evaluate against a STALE `Active` `session_status`, silently bypassing Step 0.5's I0 class-level `Draft`+`CommitIrreversible` deny for every plan node the planner submits. This has never been caught by any existing test because every shipped adversarial-planner test routes its tainted literal through a routing/content-sensitive arg, so I2's per-arg Block fires first and Step 0.5 is never reached (`DESIGN-session-trust-state.md` §11 condition 4's own documented precedence) — the bypass is real but has not yet been exercised. **This is flagged LOUDLY under §f (Cross-Cutting) as a NEW candidate X-04 the design doc should decide whether to fold into X-02/X-03's scope or explicitly accept as a residual — CONTEXT.md's X-02/X-03 text does not currently name it.**

Beyond those three, HARDEN-04 (§d) has an unusually clean path: `crates/executor/Cargo.toml` already ships a live, shipped `test-fixtures` Cargo-feature precedent (`#[cfg(any(test, feature = "test-fixtures"))]` / `#[cfg(not(any(test, feature = "test-fixtures")))]` dual-arm idiom, `sink_schema.rs:75-101`) that `crates/brokerd` (which currently has NO `[features]` section at all) can copy near-verbatim for D-09/D-10.

**Primary recommendation:** proceed with all five CONTEXT.md mechanism directions unchanged; require the design doc to (a) pin `contents`' expected-role list to include `"path"` explicitly (§e), (b) specify the new `workspace_rel`-threading plumbing for the trusted-label signal (§a), and (c) explicitly rule on the newly-found Planner-connection `session_status` staleness (§f) rather than silently inheriting it as an unstated gap.

## §a — HARDEN-01: demote-at-RequestFd

### Verified current-state (file:line, re-read this session)

- `BrokerRequest::RequestFd { path }` (`crates/brokerd/src/server.rs:996-1047`): sets `*fd_requested = true` at entry (`:1001`), resolves the path beneath the workspace dirfd via `openat2(RESOLVE_BENEATH|RESOLVE_NO_SYMLINKS)` (`:1008-1010`), appends a `fd_granted` audit Event (`:1013-1030`), passes the fd via `SCM_RIGHTS` (`:1032-1040`), and advances `last_event_id`/`last_event_hash`. **`session_status` is never read, matched, or mutated anywhere in this arm** — confirmed by direct read, not inference.
- The SOLE I1 demotion site remains `mint_from_read` (`crates/brokerd/src/quarantine.rs:284-399`), reached only from `BrokerRequest::ReportClaims` (`server.rs:1049-1135`) — a message the (possibly compromised/silent) worker chooses whether to send. `mint_from_read` demotes atomically under the same lock as its `file_read` event append (`quarantine.rs:373-398`): `update_session_status(conn, session_id, &SessionStatus::Draft)` (`crates/brokerd/src/session.rs:89-99`, a plain `UPDATE sessions SET status = ?1 WHERE id = ?2`, no `WHERE status = ...` CAS guard — one-way/idempotent by construction since `Draft` is monotonic) then an append-only `session_demoted` Event parented on the just-appended `file_read` id (`quarantine.rs:386-396`).
- `fd_requested` (`server.rs:901`, `:1001`) is exactly the kind of broker-side, per-connection, RequestFd-entry-mutated flag D-02 cites as precedent — confirmed identical shape (a `&mut bool` threaded exactly like `session_status`, `intent_provided`) already exists at the exact site D-01 wants to add a demotion.
- **CONTROL-01's clean path is real and currently untainted by fd-grant at all**: `cli/caprun/tests/s9_live_block.rs`'s `s9_control_ab` test (`:611-` onward, referenced by name in the roadmap and `CLAUDE.md`'s `MAILPIT_VERIFY_CMD` example) asserts a trusted-intent clean send exits 0, has an `email_send_succeeded` event, and zero `pending_confirmations` rows — this run's worker DOES call `RequestFd` on the CLI's designated `<workspace-file>` and reads its (fragment-free) content, yet the session stays `Active` end-to-end today because `ReportClaims` for that read reports nothing block-worthy and no untrusted claim gets minted. This is the exact test D-01/D-03's mechanism must not regress.
- **No broker-side "trusted label" plumbing exists today.** `main.rs` derives `workspace_root_dir` (the anchor directory, `WorkspaceRoot::open`, `:190-193`) and `workspace_rel` (the CLI's own `<workspace-file>` basename, `:187-189`), but `workspace_rel` is forwarded ONLY to the worker subprocess via `.env("WORKSPACE_FILE", workspace_rel)` (`:301`) — it is never passed into `run_broker_server`/`dispatch_request`. The broker currently has no in-memory notion of "this is THE file the CLI designated as this session's intent-carrying document" distinct from "any other path inside the workspace root the worker happens to RequestFd."

### Code-grounded mechanism recommendation

D-01/D-02/D-03's directions are all consistent with the code; the concrete gap is plumbing, not principle:

1. **Move (or add) the demotion to `RequestFd`'s entry, symmetric with `fd_requested = true` at `:1001`.** A new broker-derived `is_trusted_labeled(path) -> bool` check runs BEFORE the fd is opened/passed (fail-closed ordering, D-01's "commits before the fd is released"); on `false`, demote via the SAME atomic pattern `mint_from_read` already uses (`update_session_status` + causally-linked `Draft`-triggering Event, parented on the prior chain-head, mirroring `quarantine.rs:373-398`'s two-write atomicity) BEFORE `read_within`/`pass_fd` runs.
2. **`is_trusted_labeled`'s broker-derivable, non-parsing signal is a path-identity compare, not a content signal.** The natural candidate given current plumbing: does the RequestFd'd path equal the CLI's own `<workspace-file>` argument (`workspace_rel`)? This requires a genuinely NEW threading hop: `workspace_rel` (or an equivalent `Option<PathBuf>` "trusted document path") must be passed into `run_broker_server` and stored per-connection/per-session alongside `workspace_root`, exactly the way `session_status`/`intent_provided`/`fd_requested` are already threaded. **This is new plumbing, not a rewire of an existing field — Phase 27's plan should budget for it explicitly**, and the fail-closed default for `is_trusted_labeled` on any path that doesn't match (or if the value is ever `None`) must be `false` → demote (D-03's explicit fail-closed rule).
3. **D-02's pinned-doc amendment is mechanical.** `DESIGN-session-trust-state.md:80-81`'s "No other function in `brokerd` MUST be permitted to set `SessionStatus::Draft` for the I1 reason" needs one sentence added naming the new `RequestFd`-entry demotion as a second, broker-side, trusted-path-only I1 demotion site — both remain broker-only, never worker-asserted, so the anti-self-declaration invariant (`:74-87`) is preserved, not violated.
4. **`session_demoted`'s causal parent changes shape.** Today `session_demoted.parent_id == file_read.id` (`quarantine.rs:388-396`, TAINT-04's pinned causal edge). A fd-grant-time demotion has no `file_read` event yet — it must parent on `fd_granted`'s id instead (or, if `fd_granted` hasn't appended yet under the new ordering, on the CURRENT chain head immediately before it). The design doc must pin the exact causal-edge target; this is a semantic decision `DESIGN-session-trust-state.md` §5 does not anticipate.

### Risks / pitfalls / false-positive surfaces

- **Double-demotion is safe but must stay idempotent.** If a session is demoted at `RequestFd` AND later `mint_from_read` fires again on the same connection, `update_session_status`'s bare `UPDATE ... SET status = 'Draft'` (no CAS guard) is naturally idempotent — re-running it is a no-op state-wise, but a SECOND `session_demoted` Event would append (fine for audit legibility, but the design doc should state whether a second demotion event is expected/deduplicated, since `mint_from_read`'s own doc comment currently says it is the SOLE trust-flip site — that sentence becomes false the moment §a lands and must be corrected in the SAME PR, not left stale).
- **The trusted-label plumbing is new attack surface, not free.** If `workspace_rel` comparison is naively a string `==`, path normalization mismatches (e.g. worker requests `./report.txt` vs CLI's `report.txt`) could either wrongly demote a legitimate flow (safe direction, availability cost) or — if compared carelessly with a permissive normalization — wrongly treat an attacker-supplied lookalike path as trusted (dangerous direction). Must reuse the SAME `openat2(RESOLVE_BENEATH|RESOLVE_NO_SYMLINKS)`-resolved canonical form already computed for the fd-open, never a second ad hoc string compare.

## §b — HARDEN-02: authenticated audit chain (verify_chain)

### Verified current-state (file:line, re-read this session)

- Schema (`crates/brokerd/src/audit.rs:25-97`): `events` table columns `id, parent_id, session_id, event_type, actor, payload, taint, parent_hash, hash` — `hash`/`parent_hash` are plain `TEXT`, no constraint tying them to any secret. `STRICT` tables enforce column TYPE only, not content.
- `compute_event_hash` (`:243-259`): `SHA256(parent_hash.unwrap_or("") || id || session_id || event_type || payload || taint)` — an UNKEYED hash. Anyone with read access to the row's own fields can recompute it; anyone with WRITE access can edit a row and recompute a self-consistent `hash`/`parent_hash` for every descendant.
- `append_event` (`:274-315`): only ever `INSERT`s (confirmed — no `UPDATE`/`DELETE` on `events` appears anywhere outside `#[cfg(test)]` tampering helpers in this file). One extra guard: a `sink_blocked` event with zero anchors is rejected (`:283-287`, unrelated to HARDEN-02).
- `verify_chain` (`:477-539`): a recursive CTE starting from `parent_id IS NULL` (the root), walking forward via `JOIN chain c ON e.parent_id = c.id`, recomputing `compute_event_hash` per row and checking `stored_parent_hash == prev_hash` and `computed == stored_hash`. **Confirmed: this walk has no way to know how many events SHOULD exist for a session.** If an attacker with `events`-table write access `DELETE`s the LAST N rows of a session's chain (tail truncation) or restores an older DB snapshot, the recursive CTE simply terminates at the (now-shorter) true leaf and returns `true` — `found_any` only requires ≥1 row to exist. This directly confirms D-04's "anchored/monotonic chain head" requirement is not decorative: **tail truncation is currently undetectable by `verify_chain`, full stop.**
- **D-06's mutable/redactable field, precisely identified:** `blocked_literals` (`audit.rs:58-63`), a side table keyed `(event_id, arg)`, deliberately KEPT OUT of the hashed `events.payload` (only the SHA-256 digest `literal_sha256` lives inside the hashed anchor — `audit.rs:45-57`'s comment states this explicitly: "this resolves the tamper-evidence-vs-redactability conflict of storing the literal in the chain"). `redact_blocked_literal` (`:213-219`) is a real, already-shipped `DELETE`. The design doc still needs to say this explicitly (CONTEXT.md is right that it must be verified, and it is — no salted-commitment addition is needed for `blocked_literals` specifically).
- **A second mutable table exists that CONTEXT.md's D-06 framing did not name: `pending_confirmations`** (`audit.rs:86-96`). Its `state` column is mutated in place via `transition_state`'s `UPDATE ... WHERE effect_id = ?2 AND state = 'pending'` (`confirmation.rs:291-302`) — this row is NEVER inside the hashed `events.payload` chain at all (it is a wholly separate table with no MAC/hash column of its own). A DB-writer could flip a `Confirmed` row back to `Pending` (defeating at-most-once, relevant to HARDEN-03's CAS too) or delete a row outright, and `verify_chain` — scoped only to the `events` table — would not detect it. **This must be named as either a second mutable/authenticated field the MAC scheme also needs to cover, or an explicit second Accepted Residual Risk alongside D-05** — CONTEXT.md's D-06 only anticipated ONE mutable field.
- `verify_chain` callers, confirmed exhaustively via `grep -rn "verify_chain" crates/ cli/`: (1) `confirmation.rs:599` inside `confirm()`, gating BEFORE trusting anything read back from the `sink_blocked` event (Step 4.5a, `:591-603`) — this is a SEPARATE OS process (`caprun confirm`, always launched fresh, per `audit.rs:69-71`'s own doc comment: "`caprun confirm`/`caprun deny` are ALWAYS separate, later OS processes from the one that created the block ... this table is the ONLY thing that survives to resume from"); (2) `cli/caprun/src/main.rs:343`, an end-of-live-run sanity assertion inside the SAME process that ran the session. Any MAC/anchor change must keep BOTH call sites true-on-untampered-chain (no false positives) — `confirm()`'s gate in particular is the ONE place a wrong MAC key custody choice would be immediately visible (a genuinely separate process must still be able to verify).

### Code-grounded mechanism recommendation

- **Keyed MAC (e.g. HMAC-SHA256) replacing/wrapping `compute_event_hash`'s input, key held ONLY by the broker process, never written to the SQLite file.** Concretely: generate (or load) a broker-process secret at `caprun`/`brokerd` startup, held in-process memory only — never in the `sessions`/`events` schema, never in a file inside the workspace root (which the confined worker's Landlock policy could theoretically reach if misconfigured) or the audit DB path itself (a bare DB-file writer, D-04's threat model, must not be able to derive it from the DB file). **This is a genuine architectural gap today: `caprun` is single-shot-per-session (no persistent broker daemon), so a key generated fresh each process run cannot verify a PRIOR run's chain** — `confirm()` runs in a SEPARATE process from the one that appended the block, and per-run key generation would break exactly the call site that most needs the MAC (Step 4.5a). **The key MUST be a stable secret across the CLI-run and the later `confirm`/`deny` process** — e.g. derived deterministically from something both processes can independently obtain (a file alongside the audit DB but OUTSIDE the confined worker's Landlock-visible workspace root, protected by host filesystem permissions the worker never gets fd access to) or supplied via an env var/keychain the confirm/deny invocation also receives. This key-custody-across-processes requirement is NOT explicitly named in CONTEXT.md D-04 and is the single hardest design question §b must answer — pin it explicitly, do not leave "the broker holds the key" as if one broker process instance persists across the whole session lifecycle, because today it does not.
- **Anchored/monotonic head**, per D-04: a separate small value (event count, or the current head's hash) stored OUTSIDE the recomputable chain-walk path — e.g. a dedicated single-row table `chain_anchor(session_id, head_event_id, head_hash, event_count)` updated atomically with every `append_event`, itself MAC'd with the same broker key. `verify_chain`'s new contract: recompute the walk AND assert the computed leaf matches the anchor's `head_hash`/`event_count` — this is what turns tail-truncation from invisible into detected.
- **`pending_confirmations.state` needs the same authentication treatment or an explicit residual.** Recommend folding `state` (and likely `combined_digest`, already load-bearing for CONFIRM-03) into a per-row MAC using the SAME broker key, checked at `confirm()`/`deny()` entry alongside the chain-verify gate — this closes the newly-identified gap above and is a natural sibling to HARDEN-03's CAS work (same table, same phase-29-adjacent surface, though scoped to Phase 28 per the roadmap's HARDEN-02 assignment).

### Risks / pitfalls / false-positive surfaces

- **Key-custody-across-processes is the load-bearing unknown** (see above) — get this wrong and either (a) `confirm()` can never verify a chain from an earlier process (false-positive `DigestMismatch` on every legitimate confirm, breaking the shipped `s9_live_block.rs` confirm-path tests) or (b) the key ends up somewhere a bare DB-file writer (or worse, the confined worker) CAN read it, defeating D-04's entire threat model.
- **Anchor-table write must be atomic with the `events` append it anchors**, under the SAME lock/transaction discipline every other multi-write in this codebase already follows (`mint_from_read`'s two-write atomicity, `confirmation.rs`'s SEND-01 transaction) — a non-atomic anchor update reopens exactly the truncation gap it's meant to close (an attacker could `DELETE` the anchor row along with the truncated tail).
- **Migration of a pre-existing (unauthenticated) `audit.db`** — `audit.rs` already has a live precedent for schema migration (`migrate_pending_confirmations_schema`, `:99-145`, gated on `PRAGMA table_info` column presence, re-run-safe). A MAC/anchor rollout needs the same idempotent-migration discipline; an old DB with no anchor row must fail closed (never silently trusted as "already authenticated") when first opened post-upgrade.

- `crates/brokerd/src/audit.rs`: `compute_event_hash` (`:243-259`) signature change (needs the key); `append_event` (`:274-315`) call-site update; new anchor table in `SCHEMA_DDL` (`:25-97`) + a migration function mirroring `migrate_pending_confirmations_schema`; `verify_chain` (`:477-539`) rewritten to check the anchor.

## §c — HARDEN-03: Allowed-path replay CAS

### Verified current-state (file:line, re-read this session)

- The trusted `email.send` Allowed-dispatch lives in `evaluate_plan_node_and_record` (`crates/brokerd/src/server.rs:550-869`), specifically the `if matches!(decision, Allowed) && plan_node.sink.0 == "email.send"` block (`:791-866`). Confirmed: **no `PendingConfirmation` row, no CAS, no idempotency check of any kind exists on this path today.** It resolves the full arg set (`:799-814`), appends an OPAQUE `email_send_attempted` Event BEFORE opening any SMTP connection (`:820-846`, "MAJOR-4"), then invokes `invoke_email_smtp_from_resolved` (`:855-863`). The code's OWN comment at `:848-854` names the exact residual: "REPLAY RESIDUAL RISK (named, not silent) ... a replayed `SubmitPlanNode` mints a fresh `effect_id` and would send again (N submissions ⇒ N emails)."
- **`effect_id` is minted fresh, per-call, by the broker at the TOP of `evaluate_plan_node_and_record`** (`:562`, `let effect_id = Uuid::new_v4();`) — BEFORE the executor even runs. This is the critical fact D-08 gestures at but does not spell out: **a naive CAS keyed on `effect_id` (mirroring the confirm-path's `PendingConfirmation` primary key) would do NOTHING against replay**, because every resubmission of the identical `SubmitPlanNode { plan_node }` message gets a BRAND NEW `effect_id` and would sail through any `effect_id`-keyed uniqueness check. The idempotency key MUST be derived from something that is IDENTICAL across a replayed submission of the same plan node — not the broker-minted `effect_id`.
- **The exact mirror-target pattern already exists and is proven**, on the CONFIRM path's `email.send` arm (`confirmation.rs:694-738`, tagged "SEND-01" in the code): `let tx = conn.transaction()?; let affected = transition_state(&tx, effect_id, Confirmed)?; if affected == 0 { return AlreadyTerminal; } <append email_send_attempted inside tx>; tx.commit()?; <THEN open SMTP, after commit>`. `transition_state` (`confirmation.rs:291-302`) is a single `UPDATE pending_confirmations SET state = ?1 WHERE effect_id = ?2 AND state = 'pending'` — the `affected == 0` return value IS the CAS: a raced/replayed second call finds `state` already `'confirmed'` and the `WHERE` clause matches zero rows. This is a real, already-shipped, already-tested (per `confirmation.rs` test names referencing "at-most-once") pattern to mirror — but it is keyed on `effect_id`, which is stable ONLY because on the confirm path `effect_id` is looked up from a PERSISTED `PendingConfirmation` row created once at Block-time and then referenced by every later `confirm`/`deny` call by that SAME id — it is never re-minted per call. The Allowed path has no such persisted anchor to look up an `effect_id` FROM; it mints one fresh every time, which is exactly why a literal copy-paste of the confirm-path CAS does not work here without first fixing the idempotency-key derivation.
- `plan_node` itself (`runtime_core::PlanNode { sink: SinkId, args: Vec<PlanArg { name, value_id }> }`) carries no `effect_id`, no nonce, and no content-hash field today — `PlanNode`'s shape is locked (CLAUDE.md: "The broker effect path takes plan nodes from day one ... Never introduce a raw `EffectRequest`"), so any idempotency key MUST be derived, not carried as a new `PlanNode` field (a new field would also require re-verifying `check-invariants.sh` Gate 1/3 and the schema-validation discipline in `sink_schema.rs`).

### Code-grounded mechanism recommendation

- **Derive the idempotency key from the RESOLVED plan-node content, not from `plan_node` alone.** `PlanArg.value_id` is a per-connection opaque handle (`ValueId`) — two SEPARATE `SubmitPlanNode` messages carrying the exact same `value_id`s for the exact same `sink` are a textbook replay (the worker re-sent the identical message; nothing about the world changed). Candidate key: `SHA256(sink.0 || sorted(arg_name || value_id) for each arg)` computed once at the top of the `email.send` Allowed-dispatch block (`:791` onward), BEFORE the fresh `effect_id` is even relevant to the CAS. This mirrors `confirmation.rs`'s own `combined_digest` primitive shape (sorted-by-name, hash-of-pairs) closely enough to reuse conventions, though it is over `value_id`s (broker-scoped handles) rather than resolved literals — resolved-literal hashing is also viable and arguably tighter (catches the "worker mints a NEW derivation that happens to resolve to the identical literal" case D-08 explicitly says CAS does NOT need to catch, since D-08 scopes CAS to per-plan-node, not per-literal — so `value_id`-based keying is the CORRECT scope, not merely convenient).
- **New table, same shape as `pending_confirmations` but for the never-blocked path** — e.g. `sent_plan_nodes(idempotency_key TEXT PRIMARY KEY, effect_id TEXT NOT NULL, session_id TEXT NOT NULL, sent_at TEXT NOT NULL)`. The CAS becomes `INSERT INTO sent_plan_nodes (idempotency_key, ...) VALUES (...)` inside the SAME transaction as the `email_send_attempted` append (mirroring SEND-01's `tx`/`transition_state`/append/`commit()` ordering exactly) — an `INSERT` that violates the `PRIMARY KEY` constraint on a replay IS the CAS (SQLite returns a constraint-violation error rather than an affected-row count; either idiom — `INSERT OR IGNORE` + check `rows_affected() == 0`, or catching the constraint error — is a valid CAS shape, but must be picked explicitly and documented, since it's a different SQL idiom from `transition_state`'s `UPDATE ... WHERE`).
- **Commit boundary must exactly mirror SEND-01: CAS + `email_send_attempted` append commit BEFORE any SMTP socket opens** (`confirmation.rs:701-721`'s "AFTER commit — the CAS + attempt are now durable together, or neither is; only now does an SMTP connection ever open" is the exact discipline to copy into `server.rs:791-866`).

### Risks / pitfalls / false-positive surfaces

- **Keying on the broker-minted `effect_id` is the single most likely implementation mistake** (it is the visually "obvious" thing to CAS on, since that's what the confirm path does) — the design doc must state explicitly, in the same loud terms as this research, that `effect_id` is freshly minted per `SubmitPlanNode` call on the Allowed path and is therefore USELESS as a replay key.
- **D-08's scope caveat is real and must be restated, not softened.** Even a correct `value_id`-set-based CAS only catches a BYTE-IDENTICAL resubmission of the same plan node. A compromised worker that re-derives a semantically-identical-but-freshly-minted set of `value_id`s (e.g. calls `mint_from_intent`/`mint_from_derivation` again for the "same" recipient) gets a DIFFERENT idempotency key and sails through — this is D-08's "N distinct plan nodes still sends N times" in concrete mechanical terms.
- **Two concurrent Allowed dispatches for the SAME plan node** (e.g. a genuinely racing double-submit, not just sequential replay) need the transaction's `INSERT`/constraint-violation to be the actual serialization point — SQLite's default transaction isolation under WAL mode (already enabled, `audit.rs:160`) should make this safe, but the design doc should state the isolation assumption explicitly rather than leave it implicit.

### Blast radius

- `crates/brokerd/src/server.rs`: `evaluate_plan_node_and_record`'s email.send Allowed block (`:791-866`) — new idempotency-key computation + CAS-guarded transaction wrapping the existing `email_send_attempted` append.
- `crates/brokerd/src/audit.rs`: new `sent_plan_nodes` (or similarly-named) table in `SCHEMA_DDL`, with a migration function for pre-existing DBs (mirroring `migrate_pending_confirmations_schema`).
- `crates/brokerd/src/confirmation.rs`: no code change required, but its SEND-01 pattern (`:694-738`) is the literal template — the design doc should cite it by name as precedent, and a future reader diffing the two should see intentionally-parallel structure.

## §f — Cross-Cutting (X-01, X-02, X-03, and a NEW finding)

### X-01 — label continuity (broker-written files)

**Verified current-state:** No provenance/taint label of any kind is stamped on a broker-WRITTEN file today. `invoke_file_create` (`crates/brokerd/src/sinks/file_create.rs`, the sole `file.create` writer, `openat2(O_CREAT|O_EXCL|O_WRONLY, RESOLVE_BENEATH|RESOLVE_NO_SYMLINKS)`) writes bytes to disk with no accompanying xattr, sidecar file, or DB row recording "this path was written by session X with taint Y." **A write→re-read laundering loop is theoretically possible today but is NOT currently exercised by any live test** — no existing flow both (a) writes tainted `contents` via `file.create` (currently impossible anyway per §e's finding that `contents` only ever carries a trusted `"path"`-role literal) AND (b) later `RequestFd`s that same path within the same session. The loop becomes LIVE risk only once §e's content-extraction pipeline (D-12's deferred future work) actually lands — for v1.6 it is a genuinely theoretical, not yet reachable, risk, though the CONTEXT.md is correct that HARDEN-01's demote-at-RequestFd mechanism (§a) must not be defeatable by it once it does land.
**Recommendation:** since §a's `is_trusted_labeled` check (the natural home for X-01's answer) is itself new plumbing being added this milestone, the design doc should state the SAME rule for both: a path is "trusted-labeled" ONLY if it equals the CLI's designated `<workspace-file>` argument — a broker-WRITTEN file (via `file.create`) is, by this rule, automatically NOT trusted-labeled (it was never the CLI's designated document), so a later `RequestFd` on it would fail-closed-demote by default UNLESS a future milestone adds real write-time labeling. This gives X-01 a fail-closed answer for v1.6 without requiring new label-persistence machinery — record this as the mechanism, and record "broker-written files carry no explicit label; they get treated as untrusted-by-default via the same `is_trusted_labeled` fail-closed rule" as the Accepted Residual Risk framing, rather than building a new xattr/sidecar system this milestone (that bigger fix is D-12's own explicitly-deferred "full output-file provenance labeling model").

### X-02 — shared-store restart authority

**Verified current-state, and a framing correction:** CONTEXT.md's text ("a DB-writer can flip `SessionStatus` or delete a CAS row, and the change goes live at broker restart") implies an existing persistent-broker-daemon restart scenario. **Confirmed this does not describe the current architecture accurately: `caprun` has no persistent broker daemon that restarts mid-session.** `run_broker_server` is spawned fresh, once, per `caprun` process invocation (`main.rs:238-244`), and lives only for that one session's lifetime; `initial_session_status` is seeded from `create_session`'s IN-PROCESS result (`main.rs:201-203`), never re-read from the DB after that point. **The actual "restart" scenario in this codebase is `caprun confirm`/`caprun deny`/`caprun review`, which — per `audit.rs:69-71`'s own doc comment — are "ALWAYS separate, later OS processes from the one that created the block ... this table [`pending_confirmations`] is the ONLY thing that survives to resume from."** This is the real X-02 scenario: a fresh process opens `audit.db` fresh, and everything it trusts must come from that file (or a key it independently obtains, per §b). Recommend the design doc restate X-02 around this corrected framing: on EVERY `confirm`/`deny` process start, `PendingConfirmation.state`/`resolved_args`/`combined_digest` and the `sessions.status`/`events` chain are ALL recovered from disk, and (per §b's finding) `pending_confirmations` currently has no MAC/hash of its own at all — X-02's "MAC-re-verified or fail-closed re-derived" choice should apply uniformly to BOTH the `events` chain (via HARDEN-02's anchor) and `pending_confirmations` (via the newly-identified need in §b), not just the `events` table CONTEXT.md's text centers on.

### X-03 — TOCTOU / atomic ordering

**Verified current-state:** the existing Allowed-dispatch ordering ALREADY follows "authorize-before-effect": `executor::submit_plan_node` resolves its decision synchronously (`server.rs:566-572`), the resulting Event is durably appended UNDER LOCK (`:709-751`) BEFORE any sink (`file.create` or `email.send`) is invoked (`:761-866`) — this is already the "status transition commits before effect dispatch" discipline D-01/X-03 ask for, confirmed as the status quo for the SUBMIT-side ordering. **The genuine TOCTOU gap is not within one `SubmitPlanNode` call — it is `DESIGN-session-trust-state.md` §12's own "Accepted Residual Risk #1"** (a race between a concurrent `mint_from_read`-triggered demotion and a concurrent `submit_plan_node` reading a still-stale `session_status`), scoped down there as low-risk because "v0/v1.2's single-shot, single-session, effectively single-threaded-per-session process model" has "no multi-worker-per-session concurrency in scope." **That framing is now stale relative to the shipped codebase — see the NEW finding immediately below, which shows this residual is not merely a narrow theoretical race anymore, but a standing architectural gap introduced by v1.4 Phase 20's Planner-role second connection.**

### NEW finding — Planner-connection `session_status` staleness (proposed X-04, not currently named by any locked decision)

**This is the single most consequential grounding discovery of this research and should be flagged loudly to the design-doc author and the DESIGN-12 reviewer.**

Every connection `run_broker_server` accepts — the Worker connection AND every subsequent Planner connection — seeds its own per-connection `session_status` local from the SAME `initial_session_status` FUNCTION PARAMETER, cloned fresh at the top of the accept loop for EVERY connection:
- Worker connection: `let initial_status = initial_session_status.clone();` (`server.rs:202`)
- Every subsequent (Planner-eligible) connection: `let initial_status = initial_session_status.clone();` (`server.rs:231`), passed unchanged into `classify_second_connection` (`:280-343`) and from there into `handle_connection` with `ConnectionRole::Planner` (`:317-331`).

`initial_session_status` itself is `run_broker_server`'s OWN function parameter (`:149`), set ONCE by its caller (`main.rs:238-244`, using `session.status.clone()` from the IN-PROCESS `create_session()` call at session-creation time, `main.rs:201-203`). **There is no code path anywhere that re-reads `sessions.status` from the SQLite table when a SECOND (Planner) connection is accepted.** The ONLY place `session_status` is ever mutated post-creation is `ReportClaims`'s `*session_status = SessionStatus::Draft` (`server.rs:1131`) — which mutates the WORKER CONNECTION's OWN in-memory `&mut SessionStatus` local, a value that is NEVER visible to any other connection's task (no `Arc<Mutex<SessionStatus>>`, no shared cell — confirmed by the function signatures: `session_status: &mut SessionStatus` is a plain owned-per-task mutable reference, not a shared handle).

**Concrete consequence:** in the live v1.4/v1.5 adversarial-planner architecture, the normal operational order is: (1) worker connects, (2) worker `RequestFd`s + reads a hostile document, (3) worker `ReportClaims`s the extraction, demoting the session to `Draft` in the DB AND in the worker connection's own local, (4) the planner sidecar is engaged and a Planner-role connection is accepted, (5) the planner submits `SubmitPlanNode`. **At step 4, the Planner connection's `session_status` local is seeded from `initial_session_status` — the ORIGINAL `Active` value from session creation — NOT the `Draft` value the worker connection demoted to in step 3.** Step 0.5 of `submit_plan_node` (the I0 class-level `Draft` + `CommitIrreversible` deny) therefore evaluates against a STALE `Active` status for every plan node the Planner connection submits, for the entire remainder of the run.

**Why this has never been caught by an existing test, and why that is not reassuring:** every shipped adversarial-planner test (Phase 22's gate-proof test, `s9_control_ab`, etc.) routes its tainted literal through a routing-sensitive (`to`) or content-sensitive (`body`) argument, so I2's per-arg Block fires FIRST and Step 0.5 is architecturally never reached for those plan nodes — `DESIGN-session-trust-state.md` §11 condition 4 documents this precedence as INTENTIONAL ("the per-arg I2 Block always takes precedence"). This means the specific scenario Step 0.5 exists to catch — a `Draft` session submitting a `CommitIrreversible` plan node whose individual ARGS all look clean (untainted, or tainted-but-not-sensitive) — has literally never been exercised end-to-end via the Planner connection, so this staleness bug's blast radius has never been observed, only reasoned about from the code.

**Recommendation:** the design doc must explicitly decide one of: (a) fold this into HARDEN-01's scope as a required fix (make `session_status` a shared `Arc<Mutex<SessionStatus>>` across the WHOLE `run_broker_server` invocation, re-read at the top of EVERY `dispatch_request` call rather than seeded once per connection — this is the structurally-correct fix and mirrors how `planner_slot_occupied` is ALREADY a shared `Arc<AtomicBool>` for exactly the same cross-task-visibility reason, `server.rs:185`); or (b) explicitly name and accept this as a residual risk with its own justification (weaker, and harder to justify given I0's whole POINT is defending the "clean-looking values from an untrusted context" case this bug reopens specifically on the Planner path). Given CONTEXT.md's X-03 ruling already commits to "the status transition commits before effect dispatch" as the uniform atomicity rule, option (a) is the more consistent choice — a per-connection stale COPY of `session_status` is a species of the same problem X-03 names, just with a much wider blast radius (any point later in the run, not merely a narrow simultaneous race window) than the "race" framing suggests.

## Common Pitfalls (all five residuals)

- **Treating the confirm-path's `effect_id`-keyed CAS as directly reusable for HARDEN-03 without checking WHERE `effect_id` comes from on each path.** On the confirm path it is a stable, looked-up-from-a-persisted-row identity; on the Allowed path it is minted fresh every call (`server.rs:562`). Warning sign: any HARDEN-03 code that keys a new table on `effect_id` rather than a derived plan-node-content hash.
- **Assuming `pending_confirmations` is covered by HARDEN-02's chain authentication because it's "part of the audit system."** It is a SEPARATE table with no hash/MAC column of its own today (§b) — must be explicitly in scope or explicitly named a residual, not silently assumed covered.
- **Copying `body`'s `["body", "doc_fragment"]` role list pattern onto `contents` without re-tracing the actual mint site.** `contents` has no `"doc_fragment"`-equivalent untrusted vocabulary today — its only current value's role is `"path"` (§e). This is the single highest-probability "looks right, breaks the live test" mistake across all five residuals.
- **Building a persistent-broker-daemon mental model for X-02/restart scenarios.** This codebase has no such daemon; the real "restart" is a fresh `confirm`/`deny` OS process (§f/X-02 correction). Designing for the wrong restart model wastes design-doc effort and may miss the real threat surface.
- **Missing that `run_broker_server`'s per-connection `session_status` seeding is a snapshot, not a live reference**, when reasoning about "the broker resolves session_status from its own session store" (`DESIGN-session-trust-state.md` §4's language) — the CODE currently resolves it ONCE per connection from a stale outer variable, not per-call from a live store, for every connection after the first (§f NEW finding).

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
