<objective>
Implement the worker-side `LlmPlanner` proxy (a new `impl Planner` in
cli/caprun/src/planner.rs) and the minimal wiring so the worker can run the
LLM planner in place of the deterministic one. The proxy is a thin,
non-network shim: it forwards the handle IDs it is given to the off-process
sidecar over the abstract UDS and maps the reply back to a PlanNode. The
worker still submits that PlanNode through its own connection, exactly as it
does today with DeterministicPlanner — no cross-connection handle resolution,
no TCB change.

Purpose: Close PLANNER-03's "LlmPlanner is selectable in place of the
deterministic planner" while preserving the invariant that value flows AROUND
the planner (the resolving ValueStore lives on the worker's own connection).

Output: A selectable LLM planner behind Phase 20's seam; default behavior
(deterministic) unchanged.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| worker (confined) → sidecar (abstract UDS) | The proxy runs inside the confined worker; it may only forward handle IDs + slot hints, never literals, and must fail closed on a bad reply. |
| caprun main → sidecar spawn | main forwards OPENAI_API_KEY to the sidecar only; the worker never receives the key. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-21-08 | Tampering | sidecar reply → PlanNode | high | mitigate | response_to_plan_node validates sink + every value_id against the offered set; Err/transport-fail/error-signal → exit(1), no PlanNode submitted. |
| T-21-09 | Information Disclosure | proxy PlannerRequest | high | mitigate | plan()'s params are ValueId handles only (Phase-20 compile-time PLANNER-04 boundary); the proxy forwards handles + slot hints, structurally cannot forward a literal. |
| T-21-10 | Elevation of Privilege | OPENAI_API_KEY exposure | medium | mitigate | The key is passed only to the sidecar subprocess, never into the worker's env; the confined worker cannot reach AF_INET regardless. |
| T-21-11 | Spoofing | second connection to broker | low | accept | The proxy connects to the PLANNER_SOCK, not the broker; the PlanNode is still submitted on the worker's own broker connection. Phase 19's reject-second-connection guard remains the broker-side control. |
</threat_model>
