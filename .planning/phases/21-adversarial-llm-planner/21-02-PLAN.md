<objective>
Create `caprun-planner`: the out-of-process LLM sidecar. It is the ONLY place
in the workspace that depends on reqwest and reaches the network. It binds an
abstract-namespace UDS, accepts a framed `PlannerRequest`, makes a real OpenAI
tool-calling request, validates the tool-call fail-closed via Plan 01's
`parse_planner_response`, and returns a framed `PlannerResponse` (or an error
signal the worker treats as fail-closed).

Purpose: Satisfy PLANNER-03's "real LLM API call, not a stub" and PLANNER-04's
process-separation requirement — the network-calling code is genuinely
off-process from the confined worker's raw bytes, and reqwest never enters the
caprun / caprun-worker dependency surface.

Output: A separate `cli/caprun-planner` bin crate; reqwest isolated to it.

## Artifacts this plan produces
- `cli/caprun-planner/Cargo.toml` — new bin crate: deps llm-planner (path), runtime-core, tokio (net/io-util/rt/macros), serde_json, anyhow, and reqwest (default-features=false, features rustls-tls + json). The ONLY reqwest in the workspace.
- `cli/caprun-planner/src/openai.rs` — `call_openai(...)`, the request-body builder, and the tool-call-arguments extractor (pure helpers + one async HTTP call).
- `cli/caprun-planner/src/main.rs` — the abstract-UDS accept loop: framed recv PlannerRequest -> call_openai -> framed send PlannerResponse / error.
- `Cargo.toml` (root) — add `cli/caprun-planner` to `[workspace] members`.

## Cost awareness
A single plan-node request sends roughly a few hundred prompt tokens and receives
a short forced tool call (tens of tokens). At gpt-4o-mini pricing this is a small
fraction of a US cent per request (order 1e-4 dollars). Record the measured
token counts / cost in the SUMMARY once the live run in Plan 04 executes.
</objective>
