  <name>Task 1: Add SessionPolicy hardcoded-schema type to runtime-core</name>
  <files>crates/runtime-core/src/policy.rs, crates/runtime-core/src/lib.rs</files>
  <behavior>
    - A SessionPolicy that lists NO sinks denies every sink (deny-by-default, fail-closed).
    - A SessionPolicy listing `email.send` permits an `email.send` call and denies a `git.commit` call.
    - A coarse arg constraint (e.g. an allowlisted host set for an `http.request.write` `url`, or an allowlisted path prefix for a `file.write` `path`) permits a literal that satisfies the allowlist and denies one that does not.
    - An arg with no configured constraint is permitted at the arg level once its sink is allowed (sink gate is the deny-by-default axis; arg constraints only narrow further).
    - SessionPolicy round-trips through serde (Deserialize/Serialize) from a JSON document — it is loaded from a trusted file in Plan 04.
  </behavior>
  <action>
    Create `crates/runtime-core/src/policy.rs` defining `pub struct SessionPolicy` per DESIGN §5.1 — a minimal, declarative, HARDCODED-schema type (NOT Cedar, no dynamic rule engine). Fields: an `allowed_sinks` set (e.g. `BTreeSet<String>` of `SinkId` string names) that is the deny-by-default sink gate, and an `arg_constraints` map keyed by (sink name, arg name) to a coarse allowlist of permitted literal values/prefixes (hosts / paths / repos) per POLICY-01. Derive `serde::Serialize`/`serde::Deserialize` (so Plan 04 can parse it from a trusted JSON file) plus `Debug, Clone, PartialEq`. Keep runtime-core PURE — no `std::fs`/`std::io`/`std::net`/`tokio`/`async` (check-invariants Gate 2 fails the build otherwise); file reading + JSON parsing happen in the brokerd binder in Plan 04, never here.

    Add the evaluation entry point `pub fn evaluate(&self, sink: &SinkId, arg_name: &str, literal: &str) -> Result<(), PolicyDenyKind>` (or an equivalent per-arg method the executor gate calls once per plan-node arg, plus a sink-level `permits_sink(&self, sink: &SinkId) -> bool`). CRITICAL structural pin (POLICY-02 pre-condition): the evaluation surface returns ONLY permit-or-deny — `Ok(())` means permit (the executor falls through to the unchanged I2 loop), `Err(..)` means deny. There is NO variant, return value, or code path by which policy evaluation can produce an Allow-and-skip result. Deny-by-default: a sink not in `allowed_sinks` denies; a literal not matching a configured coarse allowlist denies; an unconstrained arg on an allowed sink permits.

    Provide a `SessionPolicy::default_fail_closed()` constructor returning the empty (denies-everything) policy, and a clearly-named permissive constructor for tests/Plan-03 (e.g. `SessionPolicy::allow_all()` that permits every sink with no arg constraints) — the permissive constructor is what Plan 03's POLICY-02 enforcement-order proof uses to show a permissive policy still cannot weaken an I2 Block.

    Register the module: add `pub mod policy;` and `pub use policy::SessionPolicy;` to `crates/runtime-core/src/lib.rs` alongside the existing `pub use` lines. Do NOT emit the deny reason type here if it lives in `executor_decision.rs` (Task 2) — reference it, or keep a small internal `PolicyDenyKind` in policy.rs that Task 2's `DenyReason::PolicyDeny` wraps; pick one and keep the two files' types consistent.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| trusted config → policy value | The policy schema is data authored by a trusted operator; this plan only defines the TYPE and its evaluation, not where the value comes from (Plan 04 binds it from a trusted source). |
| policy evaluation → executor decision | Policy evaluation must be structurally incapable of producing an Allow — it can only permit (fall through) or deny. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-42-01 | Elevation of Privilege | SessionPolicy evaluation surface | high | mitigate | The evaluation API returns `Result<(), _>` only — no Allow variant exists, so no policy value can grant authority it was not given (DESIGN §5.1/§5.2). Deny-by-default: an unlisted sink denies. |
| T-42-02 | Tampering | DenyReason taxonomy | medium | mitigate | PolicyDeny is a distinct machine-checkable variant (`code()=="policy_deny"`), never conflated with the I2 Block — so a policy-deny cannot masquerade as (or be masked by) an I2 Block in the audit trail (DESIGN §6 row 12). |
| T-42-03 | Information Disclosure | policy type purity | low | accept | SessionPolicy is a pure runtime-core type; no file I/O here keeps Gate 2 green. Parsing untrusted-path files is deferred to the brokerd binder (Plan 04), where the F1 containment check applies. |

No package-manager installs in this plan (v1.9 adds ZERO new crates; serde is already a workspace dependency) — no supply-chain legitimacy checkpoint required.
