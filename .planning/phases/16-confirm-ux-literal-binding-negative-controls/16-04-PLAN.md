WHY THIS IS A SEPARATE PLAN (coordinator BLOCKER-1 STRUCTURE mandate). A fresh panel traced what can
REACH a new Allowed email.send: RequestFd (server.rs:288-333) releases raw untrusted bytes and NEVER
touches session_status; the ONLY demotion is inside mint_from_read (reached only from the
worker-initiated ReportClaims path); ProvideIntent (server.rs:701-791) mints ARBITRARY worker-supplied
strings as trusted with NO once-only and NO ordering guard. So an attacker could: RequestFd → read
attacker bytes → never call ReportClaims (session stays Active) → ProvideIntent{recipient:"evil@..."}
→ all-trusted node → Allowed → with a naive dispatch, a REAL send, no block, no confirm. Mirroring
file.create is necessary but NOT sufficient — the guards below are what make the dispatch safe. This
plan is sequenced LAST (wave 3, after the wave-2 pair 16-02/16-03) and gets a distinct adversarial pass
before execution.
