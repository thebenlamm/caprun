<objective>
Build the VERIFIER + human-legibility side of CONFIRM-01/03/04 on top of Plan 16-01's producer:
(1) replace render_block_display's single-arg-or-panic display with a per-blocked-arg narration loop
(recipient AND body, verbatim/untruncated, per-arg provenance) plus the Draft/untrusted-seeded posture
statement; and (2) add confirm()'s recompute-and-compare integrity gate — recompute the combined digest
from the frozen blocked-subset literals and compare it, byte-for-byte, to the hash-chained sink_blocked
Event's combined_digest before any send, failing closed on mismatch.

Purpose: turns the digest from a write-only field into an actual integrity check (catches tampered
resolved_args or a tampered digest before an irreversible send), and makes the Block moment the demo's
climax — every blocked arg's provenance shown, nothing the human confirms is a byte they did not see.

Output: rewritten render_block_display, confirm()'s compare gate + DigestMismatch outcome, the
T-14-08 regression proof, and the CLI exit-code arm.

COORDINATOR MANDATE (T-14-08, carried from Phase 14/15 review): confirmation.rs:289-304's `assert!`
fail-closed panic on a genuinely-plural block is COMPLETELY UNTESTED today — neither existing seed
helper builds a 2+-blocked-arg PendingConfirmation (14-02-SUMMARY D5 flags this as human_judgment,
unexercised). Task 1 MUST add a test that drives the CURRENT panic path from a genuine 2-anchor fixture
(proving it fires) FIRST, then replace the guard body with real narration and update that same test to
assert the narrated output — a verified behavior replacement, not a silent deletion of untested code.

DECISION (flagged for coordinator review — resolves RESEARCH Open Question 2): blocked_arg_names is
persisted (Plan 16-01) and CONSUMED here to select the blocked subset, rather than re-deriving it from
the is_routing_sensitive||is_content_sensitive predicate a second time. Rationale: (a) DESIGN
"verifier reproducibility" says PendingConfirmation MUST persist enough to identify the subset
deterministically; (b) re-deriving with a LATER phase's changed sensitivity rules could silently
disagree with what was actually blocked at write time. The panic guard's predicate re-derivation
(14-02) is removed — the persisted names are now authoritative.

DECISION (flagged): confirm()'s compare target is the combined_digest read from the HASH-CHAINED
sink_blocked Event (via find_event_by_id on pc.session_id + pc.blocked_event_id), NOT the side-table
PendingConfirmation mirror. Rationale: an attacker who edits resolved_args in the side table could also
edit the side-table digest to match; only the DAG-payload copy is tamper-evident (breaking it breaks
verify_chain). The side-table mirror is a read convenience for display; the authoritative integrity
comparison is against the chained Event copy (DESIGN "the DAG anchor copy is the tamper-evident source
of truth").

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| persisted snapshot → confirm process | The confirm process is a separate, later OS process reading frozen resolved_args/digest; a write actor could tamper between Block and confirm. |
| human ↔ terminal display | The human's confirm authorizes an irreversible external send based ONLY on the displayed bytes. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-16-02 | Tampering | confirm() recompute-and-compare | critical | mitigate | Recompute from frozen resolved_args subset, compare to the hash-chained Event.combined_digest, fail closed with a durable event and no send on any mismatch. |
| T-16-03 | Tampering (TOCTOU) | confirm() compare→send window | high | mitigate | Feed the SAME in-memory pc.resolved_args snapshot to both the compare and the sink invocation; no DB re-read between them (DESIGN finding #5). |
| T-16-04 | Repudiation | render_block_display | high | mitigate | Verbatim, no-truncation, per-arg narration in recorded order so the human confirms exactly the bytes attested to; asserted by the long-body fixture. |
| T-16-06 | Information Disclosure | confirm() recompute path | medium | accept | The blocked literals are already displayed verbatim to the local operator by design (the confirm UX IS the disclosure); no new exposure beyond the existing single-arg contract. |
| T-16-SC | Tampering | cargo installs | n/a | accept | No new Cargo dependency this phase. No package-legitimacy checkpoint required. |
</threat_model>
