<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| persisted snapshot → confirm process | The confirm process is a separate, later OS process reading frozen resolved_args/digest; a write actor could tamper between Block and confirm. |
| human ↔ terminal display | The human's confirm authorizes an irreversible external send based ONLY on the displayed bytes. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-16-02 | Tampering | confirm() chain-verify + recompute-and-compare | critical | mitigate | Call audit::verify_chain(session) FIRST (fail closed on a broken chain, so a self-consistent literal+digest edit that breaks the hash cannot pass), THEN recompute from the frozen resolved_args subset and compare to the hash-chained Event.combined_digest, failing closed with a durable event and no send on any mismatch. Without the verify_chain call the compare is a mere consistency check between two mutable stores (MAJOR-3). |
| T-16-03 | Tampering (TOCTOU) | confirm() compare→send window | high | mitigate | Feed the SAME in-memory pc.resolved_args snapshot to both the compare and the sink invocation; no DB re-read between them (DESIGN finding #5). |
| T-16-04 | Repudiation | render_block_display | high | mitigate | Verbatim, no-truncation, per-arg narration in recorded order so the human confirms exactly the bytes attested to; asserted by the long-body fixture. |
| T-16-06 | Information Disclosure | confirm() recompute path | medium | accept | The blocked literals are already displayed verbatim to the local operator by design (the confirm UX IS the disclosure); no new exposure beyond the existing single-arg contract. |
| T-16-SC | Tampering | cargo installs | n/a | accept | No new Cargo dependency this phase. No package-legitimacy checkpoint required. |
</threat_model>
