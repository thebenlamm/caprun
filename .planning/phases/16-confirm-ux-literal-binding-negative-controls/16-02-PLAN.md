<objective>
Build the VERIFIER + human-legibility side of CONFIRM-01/03/04 on Plan 16-01's producer:
(1) rewrite render_block_display to narrate EVERY resolved_arg — blocked AND trusted-but-bound —
each marked, verbatim/untruncated, with per-arg provenance, plus the Draft/untrusted-seeded posture;
(2) add a read-only `caprun review <effect_id>` pre-decision surface so the human sees the bytes
before choosing confirm/deny (MAJOR-8); (3) add confirm()'s chain-verify + FULL-set recompute-and-compare
integrity gate (fail closed on a broken chain OR any digest mismatch before any send); and (4) parent
every confirm-phase event on the current chain head so a mismatch→retry never forks the DAG (MAJOR-7).

Purpose: turns the digest from a write-only field into an actual integrity check over the WHOLE arg
set (catches a tampered TRUSTED recipient, not just a blocked body), gives the human a real
pre-decision view (I2 is not theater), and keeps the audit DAG one unbroken chain even under repeated
mismatches.

Output: rewritten render_block_display (all-args), review(), confirm()'s chain-verify + FULL-set gate +
DigestMismatch, current_chain_head helper, the T-14-08 two-commit proof, and the CLI wiring.

BLOCKER-2 (widened domain, this round): the recompute MUST cover ALL resolved_args (ascending-name
order), matching 16-01's widened producer — a blocked-subset-only recompute would miss a rewritten
trusted recipient. And the narration MUST display ALL args (each marked BLOCKED vs trusted), no
truncation.

MAJOR-3/MAJOR-6 (chain-verify, honestly scoped): reading the digest via find_event_by_id only
DESERIALIZES — it does NOT check hashes. confirm() MUST call audit::verify_chain(conn, pc.session_id)
and fail closed BEFORE trusting the Event's digest. HONESTY (MAJOR-6, correcting Round-1 wording):
verify_chain is NOT authenticated/rooted — it recomputes hashes from the same SQLite store and nothing
pins the chain head, so an actor with events-table write access can forge the chain end-to-end. Wiring
it raises attacker cost and closes the single-store / non-recomputing multi-store tamper cases; it does
NOT make tamper-evidence "fully closed." The SUMMARY MUST state this exactly and record "the chain head
is not externally anchored" as an Accepted Residual Risk with a v2 obligation (keyed-MAC / external
head-pin) — see .planning/todos/pending. Tests MUST NOT assert a claim stronger than the mechanism.

MAJOR-7 (no fork): the current plan would parent the mismatch on blocked_event_id; confirm_granted /
confirm_denied ALSO parent there. A mismatch that leaves the row Pending, followed by a retry or a
later grant, makes blocked_event_id have two children — a fork — which this repo has empirically shown
breaks verify_chain's linear walk permanently (quarantine.rs:276-281), after which the new step-1
gate returns DigestMismatch for EVERY future confirm on an untampered DB. Fix: a current_chain_head
helper, and all confirm-phase appends parent on it.

MAJOR-8 (pre-decision surface): render_block_display is called ONLY inside confirm()/deny(), AFTER the
operator already typed the verb — so today the human "confirms" bytes they have not read. Add
`caprun review <effect_id>` (read-only) so the verbatim literals + provenance reach the human at a
point where they can still say no. Single-shot preserved (review never re-invokes submit_plan_node).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| persisted snapshot → confirm process | A separate, later OS process reads frozen resolved_args/digest; a write actor could tamper (any arg, blocked or trusted) between Block and confirm. |
| events-table write actor → verify_chain | verify_chain recomputes hashes from the same store; nothing pins the head — an actor with events write access can forge the chain (recorded residual risk). |
| human ↔ terminal display | The human's confirm authorizes an irreversible send based ONLY on the displayed bytes; `caprun review` gives a pre-decision view. |
