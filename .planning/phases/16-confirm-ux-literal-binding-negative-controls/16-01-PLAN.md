<objective>
Build the PRODUCER side of CONFIRM-03's combined-digest binding: a single SHA-256
digest over the ordered blocked-arg subset's fixed-width per-element literal_sha256s,
computed ONCE at Block time, persisted both inside the hash-chained `sink_blocked`
Event payload (tamper-evident source of truth) and mirrored into `PendingConfirmation`
(for the later confirm process to read). Also persist the ordered `blocked_arg_names`
so the confirm-side verifier (Plan 16-02) can select and re-hash exactly that subset.

Purpose: closes the "human confirms recipient, unconfirmed body ships anyway"
(B1-reincarnation) risk at the binding layer, and does so partition-bindingly so a
to/body boundary-shift cannot bypass a recompute-and-compare.

Output: `Event.combined_digest`/`blocked_arg_names`, `PendingConfirmation.combined_digest`/
`blocked_arg_names`, widened `pending_confirmations` schema + SQL, and the shared
`combined_digest()` primitive. The verifier + narration in Plan 16-02 build on these.

FIELD PLACEMENT — TARGET STATE (per the pending Round-5 reconciliation amendment).
`combined_digest` and `blocked_arg_names` live directly on `Event` (whole-Event fields),
mirroring EXACTLY the Phase-15 precedent (`derived_value_id`/`input_value_ids`/
`transform_kind` are whole-Event, not per-anchor, fields). Because `audit.rs` serializes
the WHOLE `Event` struct into the hashed `payload` column, an Event-level digest rides
inside the hash-chained payload exactly like the design's MUST requires ("inside the
hashed sink_blocked event payload → covered by the audit DAG hash chain").

DESIGN-confirm-binding.md's current prose (lines ~164-172) still says persist the digest
"INSIDE the hashed `sink_blocked` anchor payload" and speaks of "the anchor" (singular) —
phrasing that predates Phase 14's pluralization (`Vec<SinkBlockedAnchor>`, one per blocked
arg), where there is no single "the anchor" to embed a whole-set digest inside. The
coordinator's panel has VERIFIED the Event-level placement is semantically neutral (one
plan node → one sink_blocked Event → one sink; all anchors share effect_id; Event-level
retains every binding; it rides the hashed payload like Phase-15's derived_value_id) and
PRE-BLESSED reconciling the doc wording ("the SinkBlockedAnchor bytes" → "the hashed
sink_blocked Event payload") as a **Round-5 reconciliation amendment to
DESIGN-confirm-binding.md**. That amendment + its gate-record entry are being handled by
the orchestrator SEPARATELY from this plan revision; this plan builds to the Event-level
target state and does NOT re-litigate placement. Task 1 additionally corrects the now-stale
comment at `executor_decision.rs:145-146`, which claims the combined digest "rides on
`PendingConfirmation`" — the NON-hashed side table; following that literally would reopen
the old write-only-digest BLOCKER. The authoritative, tamper-evident copy is the
hash-chained `Event` payload; `PendingConfirmation` mirrors it only as a read convenience.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| side-table write actor → confirm process | The `pending_confirmations` / `blocked_literals` rows are durable state a later, separate OS process reads; a write actor could alter frozen literals between Block and confirm. |
| Block-time write → hashed audit DAG | The `sink_blocked` Event payload is the tamper-evident anchor; anything not inside it is not chain-covered. |

| T-16-01 | Tampering | combined_digest scheme (confirmation.rs) | high | mitigate | Hash each blocked arg's FIXED-WIDTH 64-hex literal_sha256 in order — never plain concatenation — so a to/body boundary shift changes at least one per-element digest (DESIGN finding #4). Asserted by the boundary-shift test in Task 1. |
