<objective>
Build the PRODUCER side of CONFIRM-03's combined-digest binding: a single SHA-256
digest over the FULL resolved_args set (BLOCKER-2 widening — every arg, in canonical
BYTE-WISE ASCENDING arg_name order, each element binding `sha256(arg_name) ‖ sha256(literal)`
as two fixed-width 64-hex digests — names AND literals, not literal-only),
computed ONCE at Block time, persisted both inside the hash-chained `sink_blocked`
Event payload (tamper-evident source of truth) and mirrored into `PendingConfirmation`
(for the later confirm process to read). Also persist the ordered `blocked_arg_names`
so the confirm-side narration (Plan 16-02) can mark which args were BLOCKED vs which
were merely trusted-but-bound.

Purpose: closes the "human confirms recipient, unconfirmed body ships anyway" risk AND
the BLOCKER-2 hole where the digest bound only the blocked subset — a side-table rewrite
of a TRUSTED, unbound arg (e.g. `to` in CONTROL-02's shape) would otherwise ship to a
new recipient the human never saw. Binding the WHOLE arg set, partition-bindingly, closes
both.

Output: `Event.combined_digest`/`blocked_arg_names`, `PendingConfirmation.combined_digest`/
`blocked_arg_names`, widened `pending_confirmations` schema + SQL + an idempotent open-time
migration, and the shared `combined_digest()` primitive. The verifier + narration in Plan
16-02 build on these.

BLOCKER-2 DOMAIN WIDENING (this round). Round-2 shipped a digest over the BLOCKED subset
only. A fresh panel found: confirm() ships the FULL resolved_args to the sink, but the
digest bound only blocked_arg_names — so a side-table write actor could rewrite a TRUSTED
arg's literal (the recipient `to` in the CONTROL-02 shape: trusted `to`, tainted `body`)
and the recompute-and-compare would still pass (the digest never covered `to`), delivering
to the attacker. RULING: the digest MUST cover ALL resolved_args in a canonical
name-ordered encoding. The Round-6 DESIGN amendment (planning-docs/DESIGN-confirm-binding.md,
now the current authoritative spec) further tightened the ELEMENT encoding: each element binds
`sha256(arg_name) ‖ sha256(literal)` — names AND literals, both fixed-width 64-hex, NOT
literal-only — ordered BYTE-WISE ASCENDING by arg_name (Rust `str` Ord over UTF-8 bytes),
with arg-name uniqueness asserted before hashing. This plan builds the
producer to that current DESIGN text; Plan 16-02 recomputes over the same domain and encoding
and displays ALL args verbatim (blocked and trusted, each marked). The DESIGN Round-6 amendment
is ALREADY APPLIED (committed) and is authoritative; this plan does not re-derive the scheme, it
builds to what the DESIGN doc now says.

FIELD PLACEMENT — TARGET STATE (per the settled Round-5 reconciliation amendment).
`combined_digest` and `blocked_arg_names` live directly on `Event` (whole-Event fields),
mirroring the Phase-15 precedent (`derived_value_id`/`input_value_ids`/`transform_kind`).
Because `audit.rs` serializes the WHOLE `Event` struct into the hashed `payload` column,
an Event-level digest rides inside the hash-chained payload exactly as the design's MUST
requires. The stale-comment correction that Round-2 asked for here is now DONE (commit
0d391ec) — do NOT touch `executor_decision.rs:143-149` again (see prohibitions).
</objective>
