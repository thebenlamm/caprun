<objective>
Build the PRODUCER side of CONFIRM-03's combined-digest binding: a single SHA-256
digest over the ordered blocked-arg subset's fixed-width per-element literal_sha256s,
computed ONCE at Block time, persisted both inside the hash-chained `sink_blocked`
Event payload (tamper-evident source of truth) and mirrored into `PendingConfirmation`
(for the later confirm process to read). Also persist the ordered `blocked_arg_names`
so the confirm-side verifier (Plan 16-02) can select and re-hash exactly that subset.

Purpose: closes the "human confirms recipient, unconfirmed body ships anyway"
(B1-reincarnation) risk at the binding layer, and does so partition-bindingly so a
to/body boundary-shift cannot bypass a recompute-and-compare.

Output: `Event.combined_digest`/`blocked_arg_names`, `PendingConfirmation.combined_digest`/
`blocked_arg_names`, widened `pending_confirmations` schema + SQL, and the shared
`combined_digest()` primitive. The verifier + narration in Plan 16-02 build on these.

ARCHITECTURAL DECISION (flagged for coordinator review — the design doc's prose is
stale here, exactly the prose-vs-implementation gap Phase 15 hit). DESIGN-confirm-binding.md
lines 164-172 say persist the digest "INSIDE the hashed `sink_blocked` anchor payload"
and its illustrative struct speaks of "the anchor" (singular). Phase 14 (D-14) already
made anchors PLURAL (`Vec<SinkBlockedAnchor>`, one per blocked arg) — there is no single
"the anchor" left to embed a whole-set digest inside. This plan resolves the gap by
adding `combined_digest` (and `blocked_arg_names`) directly to `Event`, mirroring EXACTLY
the Phase-15 precedent (`derived_value_id`/`input_value_ids`/`transform_kind` are
whole-Event, not per-anchor, fields). Because `audit.rs` serializes the WHOLE `Event`
struct into the hashed `payload` column, this still satisfies the design's MUST ("inside
the hashed sink_blocked event payload → covered by the audit DAG hash chain") while being
codebase-consistent. This is a deliberate resolution of a stale singular phrasing, NOT a
silent divergence from the design contract. `BlockedArg`'s own doc comment
(executor_decision.rs:143-146) already anticipates this: "that field does NOT live here."
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| side-table write actor → confirm process | The `pending_confirmations` / `blocked_literals` rows are durable state a later, separate OS process reads; a write actor could alter frozen literals between Block and confirm. |
| Block-time write → hashed audit DAG | The `sink_blocked` Event payload is the tamper-evident anchor; anything not inside it is not chain-covered. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-16-01 | Tampering | combined_digest scheme (confirmation.rs) | high | mitigate | Hash each blocked arg's FIXED-WIDTH 64-hex literal_sha256 in order — never plain concatenation — so a to/body boundary shift changes at least one per-element digest (DESIGN finding #4). Asserted by the boundary-shift test in Task 1. |
| T-16-02 | Tampering | Event.combined_digest placement | high | mitigate | Store the digest inside the whole-Event serialized payload (hash-chained), so a post-hoc edit to the digest or the anchor literals breaks the existing audit-DAG hash chain. |
| T-16-05 | Elevation of Privilege | blocked_arg_names / whole-set binding | high | mitigate | Persist ONE combined digest + the ordered blocked subset names, never per-arg digests — a partial-set confirm has no representation. Consumed by Plan 16-02's single-shot compare. |
| T-16-SC | Tampering | cargo installs | n/a | accept | No new Cargo dependency this phase (sha2/hex/serde_json already vendored and audited in Phase 13/14/15). No package-legitimacy checkpoint required. |
</threat_model>
