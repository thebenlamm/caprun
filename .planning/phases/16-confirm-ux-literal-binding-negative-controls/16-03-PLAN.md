<objective>
Prove the two negative controls that turn the demo from an anecdote into a controlled experiment:
CONTROL-01 (the gate is taint-driven — a trusted-intent send is Allowed with no gate, in the same run
as a doc-derived hostile block) and CONTROL-02 (a tainted body with a TRUSTED recipient still blocks —
the body dimension is not dead code redundant with the routing block).

Purpose: without CONTROL-01 the demo could be dismissed as "blocks all email"; without CONTROL-02,
CONTENT-01 is vacuously satisfied by the recipient block alone.

Output: a composed CONTROL-01 A/B test and a new CONTROL-02 body-tainted-only live fixture in
s9_live_block.rs, both Linux-verified.

SCOPE DECISION (flagged for coordinator review — resolves RESEARCH Open Question 1, the highest-
uncertainty call in this phase). server.rs's Allowed-decision sink dispatch invokes a live sink ONLY
for file.create (server.rs:575-577); there is NO email.send branch, so a trusted, never-blocked
email.send plan node reaches NO real SMTP adapter today. Phase 15's s9_live_clean_allow_path proves
the Allowed DECISION only (asserts plan_node_evaluated + absence of sink_blocked; NOT an actual send).
ROADMAP Phase 17/ACCEPT-01 explicitly owns "confirm sends exactly once via the real adapter... composes
CONTROL-01 (clean send, ungated) alongside the hostile block." This plan therefore satisfies Phase 16's
own success criterion 4 ("proceeds with NO block and NO confirm gate... in the same acceptance run as
the hostile block") at the DECISION level, and DEFERS actual trusted-path SMTP delivery to Phase 17 —
explicitly, not silently. Building the Allowed-path email.send wiring now would duplicate ACCEPT-01's
job and would touch check-invariants.sh Gate 3's mint-call-site allowlist without that gate's own
review pass. If the coordinator's panel wants real Mailpit delivery proven in Phase 16 instead, that is
a materially larger scope (new server.rs dispatch branch + Gate 3 review) and should be an explicit
re-scope decision, not an assumption baked in here.

FRAMING HONESTY (carried from Phase 15 Round-3 MAJOR-2, and from 15-04-SUMMARY's own framing note):
s9_live_clean_allow_path sources its `to`/`subject`/`body` from the TRUSTED INTENT (the CLI arg), while
s9_live_email_hostile_block derives them from a hostile doc. These are NOT a same-content A/B with "taint
flipped." The honest CONTROL-01 claim this plan asserts and documents is: "a send built from trusted
intent is Allowed and ungated; a send whose args are doc-derived is Blocked." All test names, comments,
and success-criteria language MUST use this framing — never the stronger/false "same doc, taint flipped."
