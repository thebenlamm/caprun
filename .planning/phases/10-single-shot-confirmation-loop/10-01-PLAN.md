<objective>
Create the durable checkpoint substrate for the single-shot confirmation loop: the
`PendingConfirmation` record types, the `pending_confirmations` SQLite side table, and the
broker-owned accessor functions (`insert_pending_confirmation`, `find_pending_confirmation`,
`transition_state`) plus a small `event_hash_by_id` helper. No block-time wiring, no confirm/deny
decision logic, and no CLI in this plan — only the persisted-state layer everything later builds on.

Purpose: `caprun confirm`/`caprun deny` are ALWAYS separate, later OS processes from the one that
created the block (`DESIGN-confirmation-release.md` "The Problem Being Solved"). The full
pending-action payload must therefore live in the persistent audit DB, keyed by an indexed
`effect_id`, with a one-way state machine whose terminal check is enforced in SQL. This plan is that
persistence layer.

Output: a new `crates/brokerd/src/confirmation.rs` module (types + accessors + unit tests), the
`pending_confirmations` DDL block in `SCHEMA_DDL`, and the `event_hash_by_id` audit helper.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| later confirm/deny process → persisted DB | The only channel that survives the original blocking process's exit; all decision state must be read from here, never from memory. |
| effect_id (untrusted CLI input, resolved later) → pending_confirmations lookup | An id not persisted at Block time must resolve to nothing (fail closed). |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-10-01 | Tampering/EoP | transition_state (re-transition of a terminal row) | high | mitigate | `UPDATE ... WHERE effect_id = ? AND state = 'pending'` — a terminal row matches 0 rows; the guard is in SQL, enforced on every call, not cached (CONFIRM-03). |
| T-10-07 | Tampering | Unknown/forged effect_id lookup | medium | mitigate | `find_pending_confirmation` returns `Ok(None)` for any id never persisted at Block time; downstream callers fail closed. |
| T-10-08 | Information Disclosure | resolved_args literals persisted in a side table | low | accept | Same at-rest posture as the existing redactable `blocked_literals` side table; redaction interplay is handled in Plan 03's confirm gate (Pitfall 4). |
| T-10-SC | Tampering | package installs | n/a | accept | No new npm/pip/cargo dependencies this phase (RESEARCH Package Legitimacy Audit: zero new packages) — nothing to verify. |
</threat_model>
