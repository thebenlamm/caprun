2. **Returning the minted handle (RESEARCH gap 2):** add `output_value_id:
   Option<ValueId>` to `BrokerResponse::PlanNodeDecision` (proto.rs), populated
   ONLY when `plan_node.sink.0 == "process.exec"` AND decision is `Allowed`; `None`
   for every other sink/decision (zero behavior change for file.create/email.send).
   The worker only ever gets the opaque `ValueId` handle, never the raw bytes (I1).
   `PlanNodeDecisionReduced` (planner wire) needs no change.

**mint_from_exec roots on the sink's process_exited event (does NOT append its
own).** invoke_process_exec (32-04) already appended `process_exited` and returns
its (event_id, hash, combined_output). mint_from_exec mints the value with
`provenance_chain = [that event_id]` — one event, both the exit record and the
mint root (the strongest non-stapling guarantee; the mint_from_read_anchor_identity
analog). This reconciles DESIGN §2.1/§2.4 per the RESEARCH system diagram (lines
191-199).

**Session demotion (RESEARCH A2, pinned default):** mint_from_exec does NOT demote
the session (mirrors mint_from_derivation's "does NOT demote", quarantine.rs:583-585,
NOT mint_from_read's I1 worker-report demotion). Rationale: exec taint is set
structurally, not via a worker self-report — no I1 trust-flip is implicated, and I2
Blocks a tainted exec value at the sink regardless of session status. A fresh
adversarial reviewer in Phase 34 should confirm this; it is pinned here, not left
silent.
