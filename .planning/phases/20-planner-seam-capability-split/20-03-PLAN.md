<objective>
Close the decision-oracle for the planner-role connection (DESIGN §7 / §8 DESIGN-04): a
planner connection that submits a plan node must receive only a reduced proceed/blocked signal,
never the full `Allowed`/`BlockedPendingConfirmation { anchors, literal_sha256, literal }`
detail. This enforces PLANNER-04's "handle IDs only, never literals" boundary on the RESPONSE
side, so even an adversarial planner (Phase 21) cannot use the block decision as an offline
literal-guess confirmer.

Purpose: The full decision exposes `literal_sha256` (an offline literal-guess confirmer) and,
over IPC, the plaintext `literal`. The worker legitimately needs these (it already holds the
literals; the confirmation UX depends on them). The planner has no such need and must be
structurally denied them — the DESIGN doc (CLEARED, binding) rules the reduction MUST happen at
the planner seam introduced in 20-02.

Output: A reduced response variant and the planner-role projection, with a test proving the
planner never sees anchors/digests/literals while the worker path is unchanged.
</objective>
