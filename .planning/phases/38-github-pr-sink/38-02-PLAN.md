<objective>
Build the NEW session-scoped auth-grant capability model (GITHUB-02) and the
duplicate-PR CAS primitives (GITHUB-04) in the broker, plus the `caprun grant`
CLI verb. These are the two independent gates and the replay defense that the
Plan 38-04 / 38-05 dispatch sites consume — kept in one plan so the table DDL,
the check/record helpers, the content-key derivation, and the CAS reservation
are single-sourced and cannot drift.

Purpose:
- GITHUB-02: a bearer token's authority exceeds one PR, so authorizing the token
  is a DISTINCT human action from per-effect confirm (DESIGN §4.3, FORK 3,
  mirroring the v1.4 ConnectionRole capability precedent). Absent a live grant the
  sink Denies — a bare confirm cannot create a PR.
- GITHUB-04: a replayed identical `github.pr` creates at most one PR — a
  CONTENT-derived idempotency key committed BEFORE the API call (DESIGN §4.5,
  mirroring HARDEN-03 / email.send's `sent_plan_nodes`), NEVER a PlanNode field,
  NEVER effect_id-keyed, NO clear-key-on-failure path.
Output: audit.rs capability + CAS primitives; a `caprun grant` verb.
</objective>
