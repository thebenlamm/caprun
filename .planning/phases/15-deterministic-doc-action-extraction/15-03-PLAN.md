<objective>
Wire the transform-derived mint onto the live worker->broker IPC path: a new doc-fragment claim variant, a new derived-claim request/response, and the broker dispatch arm that resolves the input handles and mints the derived value through mint_from_derivation (Plan 01). This is the broker half of EXTRACT-01's "confined extractor emits plan nodes" — the worker (Plan 04) is the confined half.

Purpose: Without a live IPC path to mint_from_derivation, the confined extractor cannot turn a hostile doc's concatenated recipient into a genuinely-threaded plan-node arg — the derived value would have to be minted worker-side (impossible; the worker never constructs ValueRecords, T-04-03) or laundered through mint_from_read (the stapling BLOCKER). This plan gives the worker the one correct channel.

Output: proto.rs additive message types + server.rs dispatch arm. Consumed by Plan 04 (worker).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| worker IPC -> broker dispatch | The worker supplies input_value_ids + a transformed literal; the broker trusts only the connection-established session identity (never a session_id in the message) and only value_ids that resolve in THIS connection's store. |
| unresolved handle -> mint | A forged/dangling/cross-connection input handle must never reach mint_from_derivation. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-15-09 | Spoofing (session) | ReportDerivedClaim | high | mitigate | The message carries no session_id; the broker uses the connection-established identity (HARD-03/T-05-03), matching SubmitPlanNode/ReportClaims. |
| T-15-10 | Tampering | input resolution | high | mitigate | Every input_value_id must resolve in the per-connection value_store; any miss fails closed (Denied), minting nothing. |
| T-15-11 | Elevation of Privilege | transform locus | high | mitigate | The broker never re-applies the transform (would put hostile-byte manipulation in the control plane, EXTRACT-01 violation); it only threads provenance from resolved inputs. |
| T-15-16 | Spoofing (of provenance origin) | ReportDerivedClaim input ORDER | critical | mitigate | A worker choosing input ORDER to place an intent_received-rooted UserTrusted handle at provenance_chain[0] is rejected: mint_from_derivation's file_read-root guard fails closed and the dispatch arm surfaces it as Denied (finding #3) — the attacker cannot pick the audited origin. |

| T-15-12 | Repudiation | causal chain | medium | mitigate | The derivation chain head is threaded into *last_event_id/*last_event_hash so verify_chain stays one linear, tamper-evident chain. |
