<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| second `caprun confirm` process → persisted audit DB | a separate, later OS process reads/writes confirmation state; MUST NOT trust any in-memory state from the original blocking process |
| human operator → sink re-invocation | the human's confirm decision authorizes exactly one sink call; MUST NOT be interpretable as broader authorization |

## STRIDE Threat Register (design-review level — this phase writes the spec that must foreclose these)

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-08-05 | Tampering / Repudiation | effect_id terminal-state check | high | mitigate | Doc MUST require the Pending/Confirmed/Denied check be read from persisted PendingConfirmation state, never memory, since confirm runs in a separate process from the block (CONFIRM-03). |
| T-08-06 | Elevation | confirm re-invocation path | high | mitigate | Doc MUST explicitly forbid confirm calling submit_plan_node a second time; confirm is a distinct, logged endorsement path that directly invokes the sink adapter with frozen Block-time args (Pitfall 3). |
| T-08-07 | Tampering | multi-arg sink reconstruction | medium | mitigate | Doc MUST require PendingConfirmation to snapshot the FULL resolved arg set at Block time, not just the one blocked arg, so confirm never re-invokes with stale/wrong non-blocked args (Pitfall 2). |
| T-08-08 | Elevation | scope of a single confirm | high | mitigate | Doc MUST state confirm releases exactly one (sink, arg, literal-digest) triple — never a standing policy or session-wide waiver (CONFIRM-02, explicitly out of scope for v1.2). |
