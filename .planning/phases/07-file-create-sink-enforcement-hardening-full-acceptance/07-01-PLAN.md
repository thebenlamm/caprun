<objective>
Make the mint invariant TRUE at the source and lay the typed-denial foundation the rest of Phase 7 builds on. Today `ValueStore::mint` (`crates/executor/src/value_store.rs:42`) accepts `vec![]` for both taint and provenance, so the "every ValueRecord carries ≥1 taint label and ≥1 provenance event id" invariant holds only by convention. This plan changes `mint` to return `Result<ValueId, MintInvariantError>` and reject empty taint or empty provenance, introduces the typed `DenyReason` enum, and moves the executor's empty-value guards UP — evaluated right after `resolve()`, BEFORE the routing-sensitivity/trust check — so a `[UserTrusted]` record with empty provenance Denies instead of reaching `Allowed` (the hole codex #5 found).

Authoritative spec: `planning-docs/TASK-mint-nonempty-invariant.md` (the change, the ripple, the `:108` allow-shape fix, Done-when) plus `DESIGN-durable-anchor-and-label-partition.md` §3 (mint invariants, guard-moves-up, typed `DenyReason`). No live behavior change — both real mint sites already pass non-empty values.

This is Phase 7's opening plan (Wave 1) because it is tiny/additive and the mint sites are freshest now. The base `DenyReason` enum it lands is consumed by 07-02 (the `ExecutorDecision` reshape) and extended by 07-04 (schema-validation variants) — which is why those depend on this plan.
</objective>
