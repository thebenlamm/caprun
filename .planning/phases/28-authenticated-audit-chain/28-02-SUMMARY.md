## Accomplishments
- Implemented `load_or_create_key(audit_path: &str, workspace_root: &Path) -> anyhow::Result<Vec<u8>>` in a new `cli/caprun/src/key.rs`: idempotent read-existing-`.key`-first custody (the Pitfall-2 cross-process-stability guard), `getrandom::fill`-backed 32-byte generation on first call, `0600` create-new file permissions (`OpenOptionsExt::mode(0o600)`), and a race-safe `AlreadyExists` fallback for concurrent first-creates.
- Implemented the F1 fail-closed startup refusal: canonicalizes the workspace root and BOTH candidate paths (the audit DB path and its `.key` sibling) — using `std::fs::canonicalize` directly where the path exists, falling back to canonicalizing the parent directory + rejoining the file name for not-yet-created paths — and refuses (`Err`, no key generated or returned, no file written) if either canonical candidate is equal to or a descendant of the canonical workspace root. An unresolvable workspace root or candidate path is itself a refusal (fail-closed, never fail-open).
- Special-cased `audit_path == ":memory:"`: returns a fresh ephemeral `getrandom`-backed key, writes no file, and skips the containment check entirely (mirrors the existing `main.rs` ":memory: fails closed" rationale — no persisted row can exist for a later process to verify against anyway).
- Added `mod key;` to `main.rs` — the module compiles as part of the `caprun` bin target; it is deliberately NOT called from either existing `open_audit_db` call site in this plan (deferred to Plan 03's run path and Plan 05's confirm/deny path, per the plan's `<objective>`).
- Wrote 3 host-side (non-Linux-gated) unit tests proving F1 refusal (`Err` + no key file written), cross-process key-byte idempotency across two independent loads, and `:memory:` ephemeral behavior — all green via `cargo test -p caprun --bin caprun -- key::`.
- Added `getrandom = { workspace = true }` as a direct `cli/caprun` dependency (already workspace-pinned at `0.4` from Plan 01; zero new dependency resolution — `Cargo.lock` only gains the existing `getrandom 0.4.3` node as a `caprun` dependent).
- Re-verified `./scripts/check-invariants.sh` (4/4 gates PASS) and `cargo test --workspace --no-fail-fast` on macOS (277 passed / 0 failed — the 274-test Plan 01 baseline plus this plan's 3 new tests, zero regressions).

## Decisions Made
- **Task-commit split correction (process, not a plan deviation):** the first implementation pass wrote both `load_or_create_key` and its unit tests into `key.rs` in one shot. Before committing, this was caught and corrected: `git reset --soft HEAD~1` (a local, unshared, unpushed commit made moments earlier by this same agent — not a shared/pushed ref, so this is not the destructive-git-prohibition's target) unwound the single combined commit, the test module was temporarily stripped for the Task 1 commit (verified `cargo build -p caprun` still green), then re-added for the Task 2 commit (verified all 3 tests green). Both commits now atomically match the plan's task boundaries.
- **Verify-command correction (Rule 3 - blocking):** the plan's Task 2 `<verify>` specified `cargo test -p caprun --lib -- key::`, which fails with `error: no library targets found in package 'caprun'` — `cli/caprun` is a bin-only crate (two `[[bin]]` targets: `caprun`, `caprun-worker`; no `src/lib.rs`). Corrected to `cargo test -p caprun --bin caprun -- key::`, which correctly resolves and runs all 3 `key::tests::*` unit tests (they live inside the `caprun` bin's own unit-test tree via the `mod key;` declaration in `main.rs`). All other plan-specified verify commands (`cargo build -p caprun`, `./scripts/check-invariants.sh`) ran exactly as written.
- F1 containment loops over BOTH the audit-DB path and the `.key` sibling path (not just the DB path) in one shared code path, per the plan's `must_haves` truth #2 and `<security_emphases>` #2 — a single `for candidate in [&audit_path_buf, &key_path_buf]` block, rather than duplicated logic.
- Chose `AlreadyExists`-as-success over a retry-then-read-back loop for the concurrent-first-create race: both processes must ultimately agree on ONE key regardless of which one physically wrote the file first, so the loser's freshly-generated (but never-persisted-or-returned-in-that-branch) bytes are simply discarded — the function then falls through to the normal "return `key`" path, but since it was this call's OWN freshly-generated `key` (not yet compared against the file), a truly concurrent race could theoretically return a losing process's in-memory bytes rather than the winner's persisted bytes for that one call. This is a narrow, documented, low-probability edge (RESEARCH itself calls concurrent creation "unlikely in practice given `caprun run` always precedes `confirm`/`deny`") — a subsequent call by the same process would always read the persisted (winner's) file. Not fixed further in this plan since it is a pre-existing risk boundary the RESEARCH doc explicitly scoped as acceptable, not a new gap introduced here; flagged here for Plan 03/05 wiring awareness.
