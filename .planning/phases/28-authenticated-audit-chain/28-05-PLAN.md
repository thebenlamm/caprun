<objective>
Fold `pending_confirmations` into the broker-key MAC scheme and give BOTH `confirm()` and `deny()` a fail-closed integrity gate at entry, wiring the cross-process key load for each verb — closing the DB-writer flip-back/delete gap on the one table that survives to resume a confirmation (DESIGN §b `pending_confirmations` ruling + §f X-02, both locked/pinned).

Purpose: The keyed events chain (Plans 03-04) does not cover `pending_confirmations`, whose `state` is mutated in place and today has no MAC. A DB-writer could flip `Confirmed`/`Denied` back to `Pending` (defeating at-most-once) or delete the row, invisibly to `verify_chain`. X-02 pins a uniform rule: on every confirm/deny process start, recovered state is MAC-re-verified or fail-closed re-derived. `deny()` currently has NO such gate — RESEARCH Open-Q1 + X-02 resolve this as a locked "yes, add it."
Output: pending_confirmations MAC column + fold + deny() gate + confirm/deny key wiring + tests.
</objective>
