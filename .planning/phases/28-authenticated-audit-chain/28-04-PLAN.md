<objective>
Add the MAC'd `chain_anchor` monotonic-head table, fold its upsert atomically into `append_event`, and extend `verify_chain` to cross-check it — turning tail-truncation from invisible (the confirmed `found_any`-only weakness) into detected, and failing closed on legacy un-anchored DBs.

Purpose: Keyed HMAC (Plan 03) stops self-consistent forgery but does NOT detect a `DELETE`-the-tail attack — the walk terminates at the shorter true leaf and `found_any` still passes. DESIGN §b D-04 pins the anchored/monotonic head as the fix. This is the tail-truncation half of HARDEN-02.
Output: `chain_anchor` table + migration + atomic upsert in append_event + verify_chain anchor assertion + fail-closed legacy handling + tests.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| in-host DB-writer → tail of the events chain | attacker deletes the last N events to hide activity |
| legacy pre-upgrade DB → post-upgrade verify | an un-anchored chain must not be trusted as authenticated |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-28-09 | Tampering / Repudiation | tail-truncation (DELETE last N events) previously undetectable | high | mitigate | MAC'd chain_anchor (head_hash + event_count) upserted atomically with each append; verify_chain asserts leaf/count == anchor — attacker cannot re-MAC the anchor without the key. |
| T-28-10 | Tampering | delete anchor row + tail together | high | mitigate | absent-anchor → fail-closed (verify_chain false), so removing the anchor does not yield a "passing" shorter chain. |
| T-28-11 | Spoofing | legacy un-anchored DB silently trusted as authenticated | medium | mitigate | migration pin: no anchor row for a session → verify_chain false (untrusted-until-re-anchored); tested. |
| T-28-12 | Tampering | non-atomic anchor write reopens truncation gap | high | mitigate | upsert folded INSIDE append_event under the same held lock (Pitfall 4), never a separate skippable call. |
</threat_model>
