<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Worker/Planner → broker `SubmitPlanNode` (Allowed decision) | A replayed authorized message must not cause a duplicate irreversible external effect (SMTP send). |
| Broker → SMTP sink | The effect must fire at most once and only after the CAS+attempt are durably committed. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-29-03 | Repudiation / DoS (side-effect exhaustion) | Allowed `email.send` dispatch (`server.rs`) | high | mitigate | Content-derived idempotency-key CAS (`INSERT OR IGNORE` into `sent_plan_nodes`) inside an atomic transaction committed BEFORE the SMTP socket opens; replay ⇒ `rows_affected==0` ⇒ suppress. Proven by `replay_cas.rs`. |
| T-29-04 | Tampering (crash-atomicity) | CAS + attempt-append ordering | medium | mitigate | Both writes in ONE `rusqlite::Transaction`, committed before any SMTP — a crash leaves both or neither, never a send without a CAS row (SEND-01 discipline; §c commit boundary). |
| T-29-05 | DoS (in-scope caveat, honest) | per-session send budget | low | accept | At-most-once is PER PLAN NODE only; a worker minting fresh distinct `value_id`s still sends N times — effects-budget/rate-limit is named future work, out of v1.6 scope (D-08). |
| T-29-SC | Tampering | npm/pip/cargo installs | low | accept | No new dependencies this phase; no install task, no legitimacy gate needed. |
</threat_model>
