  <action>
    Write `planning-docs/DESIGN-plan-executor.md` as a formal spec with MUST / MUST NOT predicates. Include these ten sections with these exact headings:

    1. "## The Problem Being Solved" — explain that if a plan calls a sink with a value that came from a tainted claim, the attacker chose the destination through data, invisibly; schema validation will not catch a well-shaped value. State the executor closes this gap for I2. Include the contrast: without the executor (split alone), instruction injection is DEFEATED but value injection is NOT DEFEATED.
    2. "## ValueNode Schema" — specify three fields: literal (the actual value, e.g. an email address), provenance (the EventId of the read Event that produced this value), taint (a list of taint labels, e.g. external.untrusted, email.raw). State every value in a plan carries all three, and taint MUST originate from the read Event — never hand-set at the sink. Present as a named field list, NOT a full fenced source file.
    3. "## PlanNode Schema" — specify fields sink (a SinkId, e.g. email.send) and args (a list of ValueNode, each carrying literal + provenance + taint). State the broker API shape, locked from day one, is submit_plan_node(session_id, plan_node) returning an ExecutorDecision. State explicitly there is NO raw EffectRequest with a plain arg map path to sinks — every effect is a PlanNode.
    4. "## Plan DAG Structure" — state v0 uses a linear PlanNode sequence (RESOLVES research assumption A2: linear is sufficient for the §9 demo; branching DAG is post-v0). Each PlanNode may reference ValueNodes produced by prior PlanNodes; taint propagates along dataflow edges; the DAG is total and acyclic so the interpreter terminates.
    5. "## Sink Sensitivity Map (v0: hardcoded)" — specify, for email.send: sensitive_args are to, cc, bcc (recipient args an attacker can redirect); non_sensitive_args are subject, body (content args, still human-reviewed for I1); effect_class CommitIrreversible; tier 4. State the v0 rule: the sink sensitivity map is hardcoded in Rust — no Cedar/policy/schema system. (RESOLVES research assumption A4: clarify which args are sensitive per v0 sink.)
    6. "## Taint Propagation Rules" — state: taint is monotonic (once a value carries a label, all derived values inherit it; labels are never removed); propagation is per-dataflow-edge (if PlanNode B consumes a tainted ValueNode output of PlanNode A, B's output inherits the taint); taint originates at read Events (a ValueNode's labels must be traceable to a specific read Event in the audit DAG); a ValueNode with taint labels but no provenance Event is invalid; the executor NEVER accepts a ValueNode whose taint was hand-set at the sink call site.
    7. "## Executor Decision Logic" — describe the deterministic, non-LLM, Rust-TCB rule in prose: for each PlanNode, for each (arg_name, value_node) in its args, if the sink's sensitivity map marks arg_name as sensitive AND the value_node has any taint label, the executor returns Block (with reason TaintedValueInSensitiveSinkArg, plus sink, arg, the literal value, the taint labels, and the provenance EventId); otherwise it returns Proceed (the broker still gates callability separately). State there is NO LLM in the enforcement path.
    8. "## Taint Provenance Requirement" — state as a dedicated section (per research Pitfall 2): taint MUST originate from a read Event recorded in the audit DAG; a ValueNode whose taint was set by the executor at sink-call time is invalid, and the §9 acceptance test considers it a failed demo. Reference the unbroken taint edge from the raw-read Event to the blocked sink argument. (RESOLVES open question 2: this doc references the audit-DAG taint-edge requirement but does NOT duplicate the Event schema — that lives with brokerd in Phase 3.)
    9. "## Literal-Value Confirmation UX" — specify that on Block, the broker surfaces the LITERAL VALUE (not a category) to the human: a prompt of the form "Proposed recipient <literal> came from untrusted content. Confirm this exact address to proceed." with Confirm/Deny. State: delivered via FAMP; no learning/auto-confirm for Tier 3+ values from tainted sources; a standing policy may permit an EXACT value/pattern but policy files cannot disable I2 globally.
    10. "## Relationship to Broker's Callability Gate & Done-When" — state the broker gate (is this sink callable at all given the Session's caps?) and the executor gate (is this specific tainted value permitted in this sensitive arg?) are independent and both must pass; neither subsumes the other. Close with the executor Done-When predicate: the doc specifies ValueNode fields, PlanNode fields, the v0 sink sensitivity map, the monotonic propagation rule, the block-on-tainted-sensitive-arg rule, and the exact literal-value confirmation prompt format.

    Briefly map to CaMeL prior art (capability metadata on values -> taint; provenance per value -> provenance EventId; policy enforcement at tool call -> Block). Reference REQ-design-plan-executor in the doc header. Do NOT create or reference any code under crates/executor.
