<objective>
Build the pure protocol substrate for the broker-performed git.push transfer: the pkt-line encode/decode + ref-advertisement + report-status parsers (RESEARCH §3), the `validate_git_refspec` value-gate and structural command-list construction that make `--force`/deletion/`+`-refspec unreachable by construction (DESIGN §1.3, RESEARCH §5), and the WG-1 frozen-IP primitive that lets ONE pinned client serve both requests of the two-request exchange without re-resolving.

Splitting protocol/pin (here) from pack-gen/transport (44-03) keeps each within context budget — git.push has materially more surface than http-write.

Output: `sinks/git_push.rs` with the protocol + structural-denial functions and the widened `resolve_and_vet`/`build_pinned_client` primitive — ready for the pack-gen + two-request driver in Plan 44-03.

  <name>Task 2: validate_git_refspec value-gate + structural command-list construction (force/delete denied by construction)</name>
  <files>crates/brokerd/src/sinks/git_push.rs</files>
  <behavior>
    - `validate_git_refspec` returns Err for: a leading `+` (force), a `<src>:<dst>` with an empty `<src>` (deletion), a bare `:<dst>`, and any `--force`/`--force-with-lease`-shaped token; it returns Ok for a plain `<src>:<dst>` or `<ref>` non-force refspec.
    - The command-list builder REFUSES (Err) to construct a receive-pack line whose `<new-oid>` is the zero-oid (delete), for any input — unreachable by construction.
    - A create (advertised old-oid absent => old-oid == zero-oid, new-oid non-zero) is ALLOWED and produces a valid create command line (WG-6: the denial keys on new-oid==zero ONLY, distinguishing it from a legitimate create).
    - The FIRST (here, only) command line carries the fixed capability set (report-status, agent) and no force capability; a subsequent flush terminates the command-list.
    - The old-oid embedded in the command line comes from the caller-supplied advertised value (the frozen info/refs advertisement), never a local-repo value.
  </behavior>
  <action>In `git_push.rs` add `validate_git_refspec(refspec: &str) -> Result<()>` mirroring `validate_write_method` (http_request.rs:183) as the exact value-gate pattern (RESEARCH A8/§5 layer 1): reject a leading `+`, an empty `<src>` in `<src>:<dst>` (and a bare `:<dst>`) = deletion, and any `--force`/`--force-with-lease`-shaped token — fail-closed. Add the command-list constructor (RESEARCH §2 step 3 / §5 layer 2) that takes {advertised old-oid, frozen new-oid, refname} and emits `pkt_line(<old-oid> SP <new-oid> SP <refname> NUL <fixed-caps>)` for the first/only command then a flush; the constructor MUST return Err when `new-oid` is the zero-oid (delete) — this is the structural denial, unreachable even via confirm (DESIGN §1.3). The capability set is fixed (report-status + agent; NO side-band-64k per Task 1, NO force capability) so a force update is not expressible by any code path. Keep the old-oid a caller-supplied parameter (the advertised value from Plan 44-03's frozen GET), never read from the local repo. Add the unit tests from the behavior block, including a force-refspec-rejected, a `:delete`-rejected, a create-allowed, and a new-oid==zero-refused test.</action>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| broker → external host | DNS resolution + the vetted `SocketAddr` that both git-push requests will pin to. |
| untrusted refspec/repo state → command-list | the refspec value + the advertised old-oid feed command-list construction; force/delete must be unreachable regardless of input. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-44-05 | Tampering | a force / deletion refspec rewriting remote history | critical | mitigate | two value-level layers: validate_git_refspec rejects `+`/empty-src/`--force` tokens; the command-list builder refuses a zero-new-oid delete by construction — unreachable even via confirm (DESIGN §1.3, §5). |
| T-44-06 | Spoofing/SSRF | DNS-rebind between the two requests moves the POST to an internal/attacker host | critical | mitigate | WG-1 single frozen `SocketAddr`: one `build_pinned_client` client for both requests, no re-resolve, redirect-none in force for the POST (DESIGN §1.5); `resolve_and_vet` reuses the shipped SSRF fail-close classifier verbatim. |
| T-44-07 | Tampering | a lying local old-oid enables force-ish behavior | high | mitigate | old-oid is a caller-supplied parameter sourced from the frozen remote advertisement (Plan 44-03), never the local repo (WG-6); a create is keyed on new-oid, not on trusting a local old-oid. |
| T-44-08 | Tampering | side-band framing hides a `ng` failure as an accepted push | medium | mitigate | side-band-64k is not advertised (simplest correct subset); the report-status parser fail-closes on anything other than a clean unpack + per-ref ok (RESEARCH §3). |
| T-44-SC | Tampering | npm/pip/cargo installs | high | mitigate | NO new crates — pure Rust pkt-line glue + reuse of the shipped reqwest+rustls(ring)+webpki-roots (HYG-01); no package-install task in this plan. |
