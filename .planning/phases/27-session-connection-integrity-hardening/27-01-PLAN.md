<objective>
Land HARDEN-01 (demote-at-RequestFd) together with the X-04 fold (shared, monotonic `Arc<Mutex<SessionStatus>>`). At the broker's `RequestFd` entry, releasing an fd for a workspace file demotes the session to `Draft` for the I1 reason unless the granted fd is inode-identical (fstat `st_dev`/`st_ino`) to the CLI-designated `<workspace-file>` — closing the gap where "reading raw untrusted bytes → draft-only" held only when the worker chose to report the read back. Simultaneously convert the per-connection `session_status` snapshot into one shared `Arc<Mutex<SessionStatus>>` re-read per `dispatch_request`, so a Planner connection accepted after a Worker demotion cannot bypass Step 0.5's I0 deny on a stale `Active` view.

Both changes touch the identical four function signatures (`run_broker_server` / `handle_connection` / `classify_second_connection` / `dispatch_request`), so the signature change lands as one atomic task that also fixes all 12 `dispatch_request` call sites (1 production at :514 + 11 test). The taint chain must be genuine: the demotion writes through the existing `update_session_status` helper and appends a `session_demoted` Event parented on the `fd_granted` Event — a real causal edge, never stapled.

Output: a broker whose fd-grant path is self-honest about I1 demotion, a single shared monotonic session-status cell, a stale-snapshot-free Planner path, and new tests proving both.
</objective>
