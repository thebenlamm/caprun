<objective>
Wire the broker-side `git.commit` sink: a new `sinks/git_commit.rs` module that
dispatches Pattern B (reusing the v1.7 `caprun-exec-launcher` + `run_launcher`),
a server.rs Allowed-dispatch arm that invokes it and mints the captured output via
the existing `mint_from_exec`, and the git config/hook neutralization that closes
the P2 RCE pitfall. This is the mechanical realization of
DESIGN-git-github-http-sinks.md §1.1, §1.4, and §1.5 (CONTEXT decisions 1, 4, 5).

Purpose: make `git.commit` an end-to-end effect — a broker-mediated, kernel-confined
`git commit` whose output taint is genuinely propagated (not stapled) and whose
config/hooks cannot execute planted code. Depends on Plan 36-01 (the executor must
recognize git.commit for the Block/classification semantics the integration tests
assert).
Output: git_commit.rs + server.rs arm + run_launcher extension + Linux-gated spawn
tests.

SCOPED DECISION (record in SUMMARY, flag for the verifier): Phase 36 wires the
Allowed path + I2 Block-detection ONLY. It does NOT wire a git.commit
confirm-release path (no invoke_git_commit_from_resolved, no prepare_git_commit,
no Step 4.75 entry-guard arm). Rationale: DESIGN §9 scopes the P33/P34
confirm-release audit-gap discipline to git.push and github.pr (CommitIrreversible)
only; ROADMAP Phase 36 success criteria do not require releasing a blocked commit;
and git.commit is a LOCAL, reversible op. A tainted `message` therefore Blocks
deterministically (the security property), and — because git.commit is absent from
the Step 4.75 confirm-release allow-list — any confirm attempt fails
closed-recoverable (row stays Pending, NO confirm_granted appended, NO audit gap).
Do NOT add git.commit to the Step 4.75 allow-list this phase.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| broker → confined exec-child (launcher→git) | broker spawns the launcher which self-confines then execve's the system git binary |
| workspace repo → git child | attacker-writable .git/config + hooks in the workspace repo are read by git |
| broker env → exec child | broker process env (secrets) must not cross into the git child or its captured output |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-36-04 | Tampering (RCE) | planted .git/config alias / .git/hooks/* | critical | mitigate | `-c core.hooksPath=/dev/null` (overrides repo-local, highest precedence) + GIT_CONFIG_NOSYSTEM=1 + GIT_CONFIG_GLOBAL=/dev/null; the broker constructs the exact `commit` argv so no alias is ever invoked. Proven by the planted-hook negative test (DESIGN §1.5, P2). |
| T-36-05 | Information Disclosure | run_launcher child env | high | mitigate | env_clear() + SAFE_EXEC_PATH + only non-secret git-neutralization vars via extra_env; the git child inherits no OPENAI_API_KEY / CAPRUN_SMTP_*. Env-clear leak test (DESIGN §1.5). |
| T-36-06 | Tampering (audit) | mint_from_exec provenance | high | mitigate | Output minted only via the existing mint_from_exec rooted on the real process_exited event (provenance_chain[0]==event id); no new mint site, no staple. Anti-staple / unbroken-DAG-edge test (DESIGN §1.4). |
| T-36-07 | Elevation (net egress) | exec-child seccomp filter | medium | accept | seccomp net-deny UNCHANGED for git.commit — a local commit needs no network; the exec-child socket(AF_INET/AF_INET6)→EPERM deny is reused verbatim, so no egress surface is added (DESIGN §1.5). |
| T-36-08 | Denial of Service | run_launcher timeout/byte-cap | low | mitigate | git.commit inherits run_launcher's wall-clock timeout + combined-output byte cap + kill_on_drop verbatim. |
| T-36-SC | Tampering | package installs | low | accept | No new npm/pip/cargo crates — reuses the shipped caprun-exec-launcher and system `git` (host deployment constant, git ≥2.30). |
