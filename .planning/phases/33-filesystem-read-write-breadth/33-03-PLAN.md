<objective>
Add the per-session `RequestFd` count limiter that makes FS-01 (multi-file read) a bounded, deliberate capability instead of "unlimited by accident."

Purpose: FS-01's multi-file read is the existing single-file `RequestFd → read_within → mint_from_read` path invoked repeatedly — no new read mechanism. But DESIGN §3.1 refuses to bless unbounded repeat calls as the default, so this plan adds an explicit per-session upper-bound counter (a resource-exhaustion guard, fail-closed deny-this-request, connection stays alive), threaded like the existing per-connection boolean guards.
Output: `MAX_REQUEST_FD_PER_SESSION` const + `fd_request_count` state in `crates/brokerd/src/server.rs`, with unit tests.

## Artifacts this plan produces
- `const MAX_REQUEST_FD_PER_SESSION: u32 = 256;` (module-level, hardcoded — matches this codebase's "security params are hardcoded constants, not config knobs" discipline)
- `fd_request_count: u32` per-connection counter threaded `&mut` through `dispatch_request`
- inline tests: `request_fd_count_limit` (deny past bound, connection open), `request_fd_repeated_reads_under_bound_succeed`
</objective>
