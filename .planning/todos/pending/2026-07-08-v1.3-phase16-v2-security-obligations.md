# v1.3 / Phase 16 — deferred v2 security obligations (recorded, NOT dropped)

**Created:** 2026-07-08 (Phase 16 round-3 replanning, from the coordinator adversarial panel)
**Status:** pending — explicitly OUT OF SCOPE for v1.3, carried forward to v2 design

These are real, panel-acknowledged residual risks/obligations. They are recorded here (and
cross-referenced from the Phase 16 plans' SUMMARYs) so no future reader mistakes v1.3's scope for
a complete story.

## 1. Demote-at-RequestFd — I1 honest scope (v2 DESIGN obligation)

`RequestFd` (server.rs) releases raw untrusted bytes to the confined worker and NEVER touches
`session_status`. The ONLY demotion site is `mint_from_read`, reached solely from the worker-initiated
`ReportClaims` path. So "reading raw untrusted bytes → draft-only" is NOT literally true at fd-release
time — the session goes Draft only when the broker MINTS untrusted taint from a REPORTED read.

- A "demote at RequestFd" fix is ruled OUT OF SCOPE for v1.3: it contradicts the pinned
  DESIGN-session-trust-state.md letter ("No other function in brokerd MUST be permitted to set
  SessionStatus::Draft for the I1 reason") and would break CONTROL-01's clean path (a benign,
  fragment-free doc read that must stay Active and send).
- v1.3 MITIGATION instead (Plan 16-04): the Allowed-dispatch is gated by BLOCKER-1's three guards
  (ProvideIntent once-only + before-RequestFd; executor non-live-state Deny; CreateSession-IPC-arm
  disabled in non-test builds), which close the attack path that made fd-release-without-demotion
  reachable to a real send.
- **DOC-01 QUALIFIER (Phase 16 SUMMARY MUST carry forward):** I1's honest scope is "draft-only triggers
  when the broker mints untrusted taint from a REPORTED read, not on fd release." Do NOT let any doc or
  demo claim "reading raw untrusted bytes → draft-only" without that qualifier.
- **v2:** design a taint-at-release model (or an equivalent) so fd release itself carries an I1
  consequence, reconciled with the CONTROL-01 clean path.

## 2. verify_chain is not authenticated / not externally anchored (v2: keyed-MAC / head-pin)

`audit::verify_chain` recomputes SHA-256 from columns in the SAME SQLite store, and `hash` is a plain
TEXT column; nothing pins the chain head. An actor with `events` write access can rewrite a payload,
recompute the hash, and fix every descendant's parent_hash/hash — `verify_chain` returns true. It is a
corruption/single-store/self-consistent-multi-store tamper detector, NOT authenticated integrity.

- v1.3 KEEPS the verify_chain call on the confirm path (Plan 16-02) — correct and fail-closed for its
  actual threat model. Claims are scoped to the mechanism (no "tamper-evidence fully closed").
- **v2:** keyed MAC over the chain and/or an externally-anchored chain head (out-of-store head pin).

## 3. Allowed-path replay has no CAS (v2: idempotency key on the trusted send)

The email.send Allowed-dispatch (Plan 16-04) has no PendingConfirmation/CAS. A replayed `SubmitPlanNode`
mints a fresh effect_id and sends again — N submissions ⇒ N emails. v1.3 records this as an
explicitly-named ACCEPTED RESIDUAL RISK (a replay requires the trusted-at-startup worker to re-submit;
the durable `email_send_attempted` ledger per attempt makes each send auditable).

- **v2:** an idempotency key / CAS on the Allowed (trusted) send path, mirroring the confirm path's
  at-most-once transaction.

## 4. Kind-aware Source label in the confirm narration (deferred UX enhancement)

`render_block_display` shows a source event id without a type label. A kind-aware label
("file_read" vs "derivation") would require threading `conn` + `find_event_by_id` through
render_block_display's three call sites (confirm/deny/review). Deferred — the v1.3 fix simply removed
the misleading hardcoded "file_read" literal so nothing is mislabeled.
